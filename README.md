# RAGRAF

**RAGRAF** — **среда проектирования операционного цифрового двойника управленческих процессов** на базе фреймворка **Σ Сигма**. Аналитик описывает регламенты как графы реакций на события от датчиков и других регламентов; платформа валидирует их по нормативным основаниям (SHACL/OWL), исполняет на симуляторе и экспортирует артефакт (Turtle / SIGMA-bundle ZIP) в исполнительный движок.

В качестве исходного **референса контракта данных** (SHACL/OWL Turtle, `/api/v1/regulations/{id}/{data,shapes}`) брался прототип Sigma по адресу `http://109.202.1.153:8958` — но это именно референс; на проде RAGRAF самодостаточен: source of truth — локальный DuckDB-store, фикстуры — seed, апстрим в коде остался как опциональная интеграция (выключена флагами `USE_FIXTURES=true` + `WRITEBACK_UPSTREAM=false`). Подмешивает GraphRAG-движок [RAGU](https://github.com/RaguTeam/RAGU) для семантического обогащения и предоставляет связанные рабочие среды:

- **Цифровой двойник** (`/twins`) — сборка процесса управления из 2+ регламентов в одно операционное представление с экспортом артефакта (Turtle / SIGMA-bundle ZIP). Здесь собирается end-to-end цепочка реагирования и передаётся в исполнительный движок.
- **Rule Flow Editor** — визуальный node-based редактор правил реагирования на React Flow (7 типов узлов, sensor-нода, DnD-палитра, валидация, версионирование). Логика **одного** регламента.
- **Constraint Editor** — табличный редактор SHACL-ограничений с инлайн-редактированием и импортом/экспортом Turtle.
- **Graph View** — обзорная карта знаний всех регламентов на Cytoscape.js (force-directed layout, фильтрация по доменам, композиционные рёбра между регламентами).
- **Sensor Library** — JSON-контракты событий для подтипов датчиков; обратный поиск «какие регламенты слушают этот датчик».

Целевая аудитория: аналитики и архитекторы семантических систем (РДФ/OWL + SHACL), эксплуатационные службы ТСЖ/НГУ/ЕДДС, операторы инцидент-центров уровня Sigma.

Полная продуктовая спецификация: [`regulation-viz-skill.md`](docs/regulation-viz-skill.md). Каталог технических навыков: [`SKILL.md`](docs/SKILL.md). Подробное описание модели Σ + триггеры + композиция регламентов: [`RULES-MANAGEMENT.md`](docs/RULES-MANAGEMENT.md). Сравнение с другими DSL: [`SKILL-D0SL.md`](docs/SKILL-D0SL.md).

## 🎯 Куда мы идём: цифровой двойник процессов управления

Это **целевая фича** платформы Σ Сигма, к которой движется проект. Остальные экраны (редактор регламента, библиотека датчиков, Flow Editor, Constraint Editor, Graph View) — **строительные блоки** одного двойника.

### Что такое «цифровой двойник управления»

Не цифровой двойник физического актива (как у Siemens MindSphere / AVEVA / GE Predix — это привязка к одному оборудованию). У нас другой класс: **operational digital twin of governance** — нормативно обоснованная модель того, **как организация реагирует на цепочку событий**.

В одном двойнике сходятся:

- **2–N регламентов** одного процесса (например — «реагирование на прорыв тепловвода»: основной регламент + регламент эскалации в ЕДДС + регламент оповещения медблока);
- **датчики** этих регламентов с JSON-контрактами событий (Sensor Library);
- **триггеры** — декларативная привязка «вход регламента ↔ источник события» (`RegulationTrigger`);
- **композиция через события** — output одного регламента служит триггером для другого (`:sourceRegulation`/`:sourceOutput`);
- **симулятор** — прогон цепочки сценариев с подсветкой fired-узлов;
- **артефакт** — Turtle / SIGMA-bundle ZIP для передачи в исполнительный движок Σ.

Аналитик собирает двойник на странице [`/twins`](/twins), экспортирует артефакт и отдаёт его в CORE Σ или регулятору как обоснованную модель управленческого процесса.

### Чем это **не** является

| Не путать с | Почему мы — другое |
|---|---|
| **n8n / Zapier / Make** | Они автоматизируют соединения сервисов. Мы фиксируем нормативно обоснованные правила реагирования с привязкой к пункту акта (`source_clause`), периоду действия и SHACL-инварианту. |
| **Camunda + BPMN** | Camunda описывает «процесс как последовательность задач». Мы — «правило как нормативная сущность» (Process = view над регламентами, не оркестрация задач). |
| **Drools / классические rule engines** | Нет онтологии (OWL/RDF/SHACL), нет визуального редактора со sensor-нодами, нет операционного представления цепочки. |
| **Цифровой двойник физического актива** (Siemens / AVEVA / GE) | Они моделируют один станок / турбину / линию. Мы моделируем **управленческий процесс**, который покрывает сотни активов и людей. |

### Сквозной сценарий аналитика

1. Открывает [`/regulations`](/regulations), создаёт три регламента одной аварийной ситуации (или импортирует существующие).
2. На каждом — задаёт параметры с порогами (SHACL min/max) и **триггеры** прямо на карточке параметра: выбирает датчик из Sensor Library или указывает «output другого регламента».
3. В Flow Editor проверяет логику отдельного регламента — sensor-узлы автоматически появляются на канвасе по триггерам.
4. На [`/graph`](/graph) видит цепочку: регламенты соединены пунктирными фиолетовыми рёбрами `triggers_regulation:<action>` через композицию.
5. Заходит на [`/twins`](/twins), создаёт двойник «Реагирование на X», добавляет в него эти три регламента, сохраняет.
6. Экспортирует SIGMA-bundle (ZIP) или объединённый Turtle (один файл) — артефакт читается стандартными OWL-инструментами (Apache Jena, Protégé) и **не зависит от RAGRAF**.
7. Передаёт артефакт в исполнительный движок Σ или регулятору как обоснованную модель управленческого процесса.

### Целевая аудитория

- **Главные инженеры АСУ ТП** — нужно описать логику реагирования промышленной системы и обосновать её для надзорного органа.
- **Аналитики цифровизации горадминистраций** — нужно собрать регламенты ЕДДС/ЦУКС/коммунальных служб в одну модель и связать с датчиками.
- **Методологи отраслевых регуляторов** — нужно описать «как должна реагировать система» в верифицируемой форме.
- **CDO компаний КИИ** — нужно зафиксировать SHACL-инварианты на критически важной инфраструктуре с трассировкой до нормативного акта.

Это **не** аудитория n8n или Zapier — другой класс задач.

## 💾 Установка на ноутбук аналитика (Windows / macOS)

RAGRAF — это **локальное self-contained ПО**. Никаких облачных зависимостей,
никакие данные не уходят за пределы машины. Сценарий внедрения у заказчика:

1. Аналитик мэрии / университета скачивает 1-2 файла с GitHub.
2. Запускает двойным кликом → откроется консольное окно.
3. Скрипт проверяет 3 бесплатные программы (Git, Python 3.11+, Node 18+),
   качает свежий код с GitHub в `~/RAGRAF/` (или `%USERPROFILE%\RAGRAF\` на
   Windows), ставит зависимости, собирает фронтенд, запускает backend,
   открывает браузер.
4. Все данные пишутся локально в `~/RAGRAF/data/` — этим каталогом аналитик
   единолично владеет.

**Каждый последующий запуск** скрипт сам проверяет обновления на GitHub и
предлагает обновиться (Y/N) с показом списка коммитов. Данные не затрагиваются
при обновлении кода — изменения регламентов и аудит-журнал переживают
переустановки.

| Платформа | Файл для скачивания | Инструкция |
|---|---|---|
| **Windows 10/11** | [`installer/start-ragraf.bat`](installer/start-ragraf.bat) | [`installer/INSTALL-WINDOWS.md`](installer/INSTALL-WINDOWS.md) |
| **macOS** (Intel/Apple Silicon) | [`installer/start-ragraf.command`](installer/start-ragraf.command) + [`installer/ragraf-mac.sh`](installer/ragraf-mac.sh) | [`installer/INSTALL-MACOS.md`](installer/INSTALL-MACOS.md) |
| **Linux** | `git clone && cd backend && python3 -m venv .venv && ...` | см. [docs/ARC.md](docs/ARC.md) или используйте `installer/ragraf-mac.sh` как образец |

## Что умеет сервис

### 1. Управление регламентами по доменам

Регламенты сгруппированы по бизнес-доменам — это устраняет «кашу» в графе и даёт каждому домену свою визуальную идентичность (иконка Lucide, акцентный цвет, подсказка):

| Домен | Иконка | Цвет | Описание | Регламенты |
|-------|--------|------|----------|------------|
| **Теплоснабжение** | 🔥 `Flame` | orange | Тепловые узлы, давление, температура, аварийные перекрытия | `pressure-diameter`, `heat-inlet-breach` |
| **Управление ЖКХ** | 🏢 `Building2` | blue | ТСЖ, общежития, придомовая территория, сезонные риски | `roof-snow-fencing`, `dormitory-flood` |
| **Безопасность кампуса** | 🛡 `ShieldAlert` | rose | Серверные НГУ, эскалация в охрану / 01-101-112 / ЕДДС, контроль доступа парковки | `thermal-incident-server`, `nsu-parking-anpr` |
| **Городская экология** | 🌿 `Leaf` | emerald | Качество воздуха, PM2.5, НМУ, оповещение уязвимых групп | `air-quality-smog-trap` |
| **Ситуационный центр** | 🛟 `LifeBuoy` | amber | ЕДДС, приём 112, эскалация в ЦУКС, режимы повседневной / повышенной готовности / ЧС, мониторинг АДПИ GSM | `koltsovo-edds-call-112`, `koltsovo-edds-mode-escalation`, `koltsovo-edds-cuks-report`, `koltsovo-edds-adpi-monitoring` |

Каждый регламент содержит: имя, дату, набор параметров (`<name>` + парное `<name>Deviation`), SHACL-форму с границами `sh:minInclusive`/`sh:maxInclusive`, рекомендацию (многострочный текст), стартовый Rule DSL для flow-редактора.

### 2. Реальные данные из 11 регламентов

Из коробки загружено **11 регламентов в 5 доменах** (включены при `USE_FIXTURES=true`):

#### Теплоснабжение

- **`pressure-diameter`** — Регламент на допустимые параметры давления и диаметра трубопроводов водоснабжения.
  *Источник:* `Rules-Management.pdf` (Постановление № 001/2023). Параметры: давление 20.5 ± 1.5 атм, диаметр 5.0 ± 0.2 см. Триггер: выход за допустимый диапазон → проверка герметичности.

- **`heat-inlet-breach`** — Регламент при прорыве теплового ввода.
  *Источник:* `~/demo-sigma-main/src/config/regulations.yaml` (сценарий `heat-inlet-breach`). Параметры: давление 4.0 ± 0.5 атм, температура подачи 70 ± 10 °C, скорость падения давления 0.0 ± 0.2 атм/мин, время подтверждения обходчика 7 ± 3 мин, окно оповещения медблока 15 ± 5 мин. Эскалация: обходчик → smart-valve → аварийная бригада + оповещение реанимационного медблока.

#### Управление ЖКХ

- **`roof-snow-fencing`** — Регламент огораживания придомовой территории при риске схода снега и падения сосулек (ТСЖ).
  *Источник:* пользовательский ТСЖ-кейс. Параметры: температура −2 ± 2 °C, скорость подъёма 0.5 ± 0.5 °C/ч, толщина снега 3 ± 2 см, длина сосулек 5 ± 5 см, заблаговременность SMS 4 ± 1 ч. Триггер: резкий выход температуры с минуса на плюс → SMS подрядчику и управляющему, монтаж сигнальной ленты, наряд на очистку кровли.

- **`dormitory-flood`** — Регламент при ночной протечке в жилом блоке общежития.
  *Источник:* `demo-sigma-main` (сценарий `dormitory-flood`, регламент управления кампуса НГУ). Параметры: уровень воды 0 ± 0.5 см, расход стояка 0.5 ± 0.3 м³/ч, начало ночного режима 22.0 ± 0.5 ч, время прибытия коменданта 15 ± 5 мин, порог эвакуации 5 ± 1 см. Эскалация: комендант на осмотр (минимизация контакта с жильцами) → автоматическое перекрытие стояка после подтверждения → эвакуация в холл 1-го этажа при критическом уровне.

#### Безопасность кампуса

- **`thermal-incident-server`** — Регламент при термическом инциденте в серверной НГУ (перегрев / задымление).
  *Источник:* `demo-sigma-main` (сценарий `thermal-incident`) + Инструкция НГУ — Приложение №6 к приказу № 721-3 от 26.03.2024. Параметры: температура серверной 22 ± 4 °C, концентрация дыма 0 ± 50 ppm, расход охлаждения 800 ± 200 м³/ч, время подтверждения старшего ИТ 5 ± 2 мин, время эскалации в ЕДДС 10 ± 3 мин. Уровни: warning → ИТ-служба; elevated → охрана НГУ + дежурный; critical (T > 35 °C либо дым > 200 ppm) → 01/101/112 + ЕДДС.

#### Городская экология

- **`air-quality-smog-trap`** — Регламент при экологической ловушке: безветрие + загрязнение PM2.5.
  *Источник:* `~/NSK_OpenData_Bot/config/rules/ecology_rules.yaml` (`risks.smog_trap`, `risks.pdk`) + `src/ecology_cache.py`. Параметры: ветер 3.0 ± 1.5 м/с (порог штиля 1.5), PM2.5 10 ± 10 мкг/м³ (норматив ВОЗ 35), часов выше ПДК 0 ± 4, заблаговременность SMS уязвимым 6 ± 2 ч, целевое снижение выбросов 17.5 ± 2.5 %. Эскалация: рекомендации жителям (очистители, маски FFP2) → SMS астма/ХОБЛ/дети/пожилые → режим НМУ + предписания предприятиям 15–20% → экстренное оповещение населения + школы/детсады при ПДК ВОЗ.

#### Транспорт и парковки (под доменом «Безопасность кампуса»)

- **`nsu-parking-anpr`** — Регламент контроля доступа на парковку НГУ.
  *Источник:* ОРД НГУ 2024 + контракт ANPR-системы. Параметры: время распознавания номера, лимит стоянки, фолбэк-звонок при отказе ANPR, месячный отчёт по нарушениям. Эскалация: автоматический шлагбаум по белому списку → телефонный фолбэк в КПП при отказе камеры → еженедельный отчёт уполномоченному.

#### Ситуационный центр / ЕДДС

Четыре регламента оцифрованы из реального **«Положения о ЕДДС МКУ "СВЕТОЧ"»** (р.п. Кольцово, Новосибирская область) — Приложение 1 к приказу № 15 от 04.12.2020. Базовый сценарий: оператор-диспетчер ЕДДС принимает поток вызовов 112, переводит дежурную смену между режимами функционирования (повседневный → повышенной готовности → ЧС), эскалирует в ЦУКС ГУ МЧС России по НСО, параллельно мониторит автономные пожарные извещатели АДПИ GSM в социальных объектах. Это самый плотный кейс event-driven композиции в RAGRAF: один и тот же вызов на 112 может стать триггером для всех трёх остальных регламентов одновременно.

- **`koltsovo-edds-call-112`** — Регламент приёма вызова на единый номер 112.
  *Источник:* §6 «Порядок работы при поступлении вызова 112» Положения о ЕДДС. Параметры: время ответа оператора 8 ± 2 с (норматив 10 с), время первичной обработки 75 ± 15 с, время передачи в профильную службу 5 ± 1 мин, время дозвона при перегрузке 30 ± 10 с, рабочая нагрузка диспетчера 4 ± 1 вызовов/час. Эскалация: приём → классификация (медицина/пожар/полиция/коммуналка) → передача в профильную ЕДС → контрольный звонок заявителю → запись в журнал. Выход: `verdict.classification` — триггер для остальных регламентов ЕДДС.

- **`koltsovo-edds-mode-escalation`** — Регламент перевода ЕДДС в режимы повышенной готовности и ЧС.
  *Источник:* §4 «Режимы функционирования» Положения о ЕДДС + Постановление Правительства РФ № 794. Параметры: пороговое количество одновременных инцидентов для перехода в повышенную готовность 3 ± 1, для ЧС 5 ± 1, время оповещения главы городского округа 15 ± 5 мин, время сбора дежурной смены 30 ± 10 мин, время оповещения населения через РАСЦО 5 ± 2 мин. Триггер на вход: композиция от `koltsovo-edds-call-112` (verdict.classification = «массовое»). ### 8.1. 🧮 Опыт математиков в булевой логике — трёхзначная Клини

> **TL;DR:** в логике формул RAGRAF мы взяли семантику из D0SL — академического DSL
> для онтологически чистых регламентов. Третье значение `unknown` решает
> «уверенный false при offline-датчике» и приводит RAGRAF в соответствие с
> SHACL-семантикой `Violation` / `Warning` / unknown conformance.

#### Откуда заимствование

В исследовании [SKILL-D0SL.md § 8.1](docs/SKILL-D0SL.md) мы разбирали альтернативный
подход к языкам процессов — **D0SL** (Domain-Oriented Specification Language)
от группы Института систем информатики СО РАН (А.С. Клещёв и соавторы). D0SL
строит регламент как **онтологическую спецификацию**, а не процедурный граф —
и одно из его ключевых отличий от RAGRAF — **трёхзначная логика Клини**:
вместо `true`/`false` есть три значения **`true` / `false` / `unknown`**,
где `unknown` означает «недостоверно» — данные отсутствуют, или тип не
сматчился, или измеритель offline.

Это идея 1938 года от Стивена Клини (Stephen Kleene) и Яна Лукасевича — двух
математиков, обозначивших фундаментальный пробел в булевой логике для систем
с неполнотой данных. С тех пор Kleene 3-valued logic используется в SQL
(`NULL` — это и есть `unknown` в WHERE-условиях), в SPARQL/SHACL (где
`Violation` ≠ `Conformance` ≠ unknown), в формальной верификации.

#### Что было не так

Раньше formula-нода `pressure > 20 && temperature > 50` при offline
pressure-датчике возвращала **`false`** (потому что `None > 20` ≈ TypeError →
catch → fallback). Оператор СЦ видел вердикт «всё ок» — **уверенно false** —
хотя по факту RAGRAF просто *ничего не знал* о давлении. Это классический
**false negative**: пропуск аварии из-за технического сбоя.

#### Что сделали

Теперь formula evaluator различает три исхода:

| Результат | Семантика | Поведение downstream |
|---|---|---|
| `true` / truthy | условие выполнено, есть данные | нода fired, сигнал идёт дальше |
| `false` / falsy | условие НЕ выполнено, есть данные | нода не fired |
| `None` (unknown) | **недостаточно данных для вывода** | нода не fired, в trace явное `unknown (нет данных: pressure, temperature)` |

Операции распространяют `None` по правилам Клини:

```
None and True  = None      None and False = False   (short-circuit известен)
None or  True  = True      None or  False = None    (short-circuit известен)
not None       = None
None + 1       = None      (любая арифметика)
None > 5       = None      (числовое сравнение)
None == None   = True      (явный тест «есть ли данные»)
```

Reference: [`formula_eval.py`](backend/app/services/formula_eval.py),
[`FORMULA-SPEC.md` § «Kleene unknown»](docs/FORMULA-SPEC.md).

#### Чем полезно проекту

**1. Честные отчёты вместо ложно-уверенных.** Дежурный оператор СЦ видит
`unknown (нет данных: pressure)` — и понимает, что нужна диагностика
датчика, а не «всё в норме». Это снимает класс ошибок «потеря сигнала
маскируется зелёным светом».

**2. Совместимость с SHACL-семантикой.** SHACL уже различает три состояния:
`sh:Violation` (нарушение) / `sh:Warning` (предупреждение) / отсутствие
проверки. Наши `compare`/`formula` теперь говорят на том же языке.
Цифровой двойник процесса в Turtle и flow-executor дают согласованные
вердикты.

**3. Защита для цифрового двойника управления.** Когда RAGRAF выйдет в
прод для НГУ-кампуса или ЕДДС, реальные ETL-каналы НЕ всегда стабильны:
датчики уходят в офлайн, ETL-пайплайн задерживает сэмпл, оборудование
заменяется. Без unknown-семантики каждый такой сбой ложно говорил бы
«норма», провоцируя пропуск инцидента. С unknown-семантикой регламент
**не зависает** (как было в одном из багов раньше), но и **не выдаёт
вердикт без оснований**: level=0 + явная пометка в trace.

**4. Идиомы для гибкости.** Разработчик регламента может явно выбрать
политику «по умолчанию» для unknown:

```python
# Offline = «не срабатывает» (paranoid mode для downstream)
pressure > 20 if pressure != None else False

# Offline = «срабатывает» (paranoid mode для оператора —
# лучше зря оповестить, чем пропустить)
pressure > 20 if pressure != None else True

# Дефолт для time-series когда мало истории
rate("temp", "1h") or 0
```

Это даёт **языковую честность**: интерпретация неполноты данных
прописана **в формуле**, а не зашита в executor'е дефолтом. На review
регламента понятно, как обрабатываются пропуски.

**5. Связь с другими доменами математики.**
SQL'ная `NULL`-семантика, нечёткая логика, теория вероятностей,
формальная верификация — все используют unknown как первоклассное
значение. Это означает, что RAGRAF может расширяться в этих направлениях
без слома модели формул: например, добавить `probability(p > 20)`
для байесовских регламентов или `definitely(p > 20)` для модальной
логики.

**Источники и приоритеты при чтении `GET /api/regulations/{id}`:**

1. DuckDB store (authoritative — `regulations.duckdb` создаётся автоматически при первом старте, сидится из фикстур)
2. Локальная фикстура (для не-редактировавшихся регламентов при `USE_FIXTURES=true` — дефолт)
3. Upstream Sigma API (**опционально**; срабатывает только если `USE_FIXTURES=false` и в store пусто — по умолчанию выключено, в проде не используется. Оставлен как путь обратной интеграции с прототипом-референсом).

**Запись `PUT /api/regulations/{id}`:**
- Всегда пишем в DuckDB (создаётся snapshot в `regulation_history`)
- Если `WRITEBACK_UPSTREAM=true` — дополнительно сериализуем в Turtle через `regulation_to_turtle()` и шлём `PUT /api/v1/regulations/{id}/data` в upstream

### Схема DuckDB

`backend/data/regulations.duckdb` (3 таблицы, создаются в lifespan при первом старте):

```sql
regulations (source_id PK, name, domain, date, version, status,
             recommendation, recommendation_priority, updated_at)

parameters  (PRIMARY KEY (source_id, id), name, datatype, ref_value,
             deviation, unit, min_inclusive, max_inclusive, position)

regulation_history (version_id PK, source_id, snapshot JSON,
                    created_at, author, comment)
```

Почему DuckDB, а не SQLite: тот же file-in-file формат без сетевой инфраструктуры, native JSON-тип для snapshot-ов истории, быстрый аналитический COUNT(*) на больших списках. Mature Python binding (`duckdb>=1.2`).

## Запуск

### Одна кнопка

- **macOS / Linux:** двойной клик по `start.command` (или `./start.command` в терминале)
- **Windows:** двойной клик по `start.bat`

Скрипт:
1. Создаёт `backend/.venv` и ставит зависимости (если ещё нет)
2. Делает `npm install` во `frontend/` (если ещё нет)
3. Копирует `backend/.env.example` → `backend/.env`
4. Поднимает FastAPI (`:8000`) и Vite (`:5173`), ждёт пока оба ответят
5. Открывает `http://localhost:5173` в браузере

Лог-файлы — в `logs/backend.log` и `logs/frontend.log`. Остановка — `Ctrl+C` (закрывает оба сервиса).

Флаги:

```
--no-open               не открывать браузер
--backend-only          только backend
--frontend-only         только frontend
--reset-deps            переустановить .venv и node_modules
--port-back=8001        другой порт backend (фронт пробросит туда же)

```

```bash
# Backend
cd backend
python3 -m venv .venv && .venv/bin/pip install -r requirements.txt
cp .env.example .env
.venv/bin/uvicorn app.main:app --reload --port 8000

```

## LLM-модели и Ollama

RAGRAF Студия аналитика использует локальную LLM через [Ollama](https://ollama.com) — никакие данные не уходят в облако, всё работает на ноутбуке.

### Установка Ollama

| OS | Команда |
|---|---|
| **macOS** | <https://ollama.com/download/Ollama-darwin.zip> (или `brew install ollama`) |
| **Linux** | `curl -fsSL https://ollama.com/install.sh \| sh` |
| **Windows** | <https://ollama.com/download/OllamaSetup.exe> |

После установки Ollama стартует фоновым сервисом на `http://localhost:11434` — это значение по умолчанию в `OPENAI_BASE_URL`.

### Где Ollama хранит модели

По умолчанию:

| ОС | Путь |
|---|---|
| **macOS** | `~/.ollama/models/` |
| **Linux** | `/usr/share/ollama/.ollama/models/` (если установлено через скрипт) или `~/.ollama/models/` |
| **Windows** | `C:\Users\<user>\.ollama\models\` |

Структура:
- `blobs/` — веса моделей (тяжёлые `.gguf` бинарники, до нескольких ГБ каждая)
- `manifests/` — метадата (теги, размеры, версии)

Сменить расположение можно через переменную окружения `OLLAMA_MODELS`:

```bash
export OLLAMA_MODELS=/Volumes/External/ollama-models
ollama serve   # перезапустить сервис чтобы подхватил
```

### Какие модели нужны RAGRAF

| Назначение | Модель | Размер | Tag |
|---|---|---|---|
| **Точная LLM** (default) | qwen2.5:7b-instruct | 4.7 ГБ | `qwen2.5:7b-instruct-q4_K_M` |
| **Быстрая LLM** (опционально) | qwen2.5:3b-instruct | 1.9 ГБ | `qwen2.5:3b-instruct-q4_K_M` |
| **Embedder** (обязательно) | bge-m3 | 1.2 ГБ | `bge-m3:latest` |

Скачать всё разом:

```bash
ollama pull qwen2.5:7b-instruct-q4_K_M
ollama pull qwen2.5:3b-instruct-q4_K_M
ollama pull bge-m3:latest
```

**Минимум для запуска**: только `qwen2.5:7b` + `bge-m3` (~6 ГБ на диске).
Быструю 3b можно докачать позже из UI — в правой панели Студии есть переключатель моделей; если 3b не установлена, рядом появится плашка с командой `ollama pull`.

### Управление моделями

```bash
ollama list      # что установлено + сколько весит
ollama ps        # какие сейчас загружены в RAM (горячие)
ollama rm <tag>  # удалить модель (освободит место на диске)
ollama show <tag>   # детали модели (context length, family, …)
```

### Прогрев и выгрузка из RAM

В правой панели Студии аналитика рядом со строкой `LLM:` есть кнопка **«прогреть» / «в RAM»** — кликом загружаешь модель в память Ollama заранее, чтобы первый ответ был без cold-start. Под капотом: `POST /api/generate` с `keep_alive: -1`. Повторный клик выгружает модель (`keep_alive: 0`) и освобождает RAM.

Это удобно когда сидишь подолгу в Студии (модель уже в памяти, отвечает сразу) или наоборот закрываешь Студию и хочешь освободить 5 ГБ под другие задачи.

### Настройка через `.env`

```dotenv
LLM_PROVIDER=ollama
OPENAI_BASE_URL=http://localhost:11434/v1
OPENAI_API_KEY=ollama
RAGU_LLM_MODEL=qwen2.5:7b-instruct-q4_K_M
RAGU_EMBED_MODEL=bge-m3
RAGU_ENABLED=true
EMBEDDINGS_ENABLED=true
```

`OPENAI_API_KEY=ollama` — это просто dummy-значение, локальная Ollama его игнорирует. `RAGU_LLM_MODEL` задаёт «модель по умолчанию» — её можно переопределить на каждый запрос через UI.

## Multi-provider LLM (Cerebras / Groq / OpenAI / Ollama)

Backend провайдер-агностичный. Переключение делается одной переменной
`LLM_PROVIDER` без правок кода — фронт автоматически подтягивает каталог
моделей из `/api/sandbox/llm-info` и рисует их в правой панели Студии.

| Провайдер | Когда выбирать | Скорость | Цена |
|---|---|---|---|
| `ollama` | Локально, без интернета, чувствительные данные | ~5–15 т/с | бесплатно |
| `cerebras` | Прод: дешёвый Wafer-инфер, ~1500 т/с | очень высокая | free-tier есть |
| `groq` | Альтернатива Cerebras, бесплатный лимит | 250–750 т/с | free-tier |
| `openrouter` | Доступ к 100+ моделям через один API | средняя | pay-per-token |
| `openai` | Дорогие премиум-модели (GPT-4o, o1) | средняя | $$$ |
| `mock` | Тесты и CI, без реальных запросов | 0 | бесплатно |

Поддерживаемые модели (на 2026-05) поднимаются автоматически из живого
`/v1/models` каждого провайдера + жёсткие presets в [`frontend/src/components/sandbox/llmModels.ts`](frontend/src/components/sandbox/llmModels.ts). Для Cerebras это
`qwen-3-235b-a22b-instruct-2507`, `gpt-oss-120b`, `zai-glm-4.7`, `llama3.1-8b`.

### Гибридный режим (cloud chat + local embeddings)

Если хочется быстрый cloud chat, но эмбеддинги держать локально (нет
бесплатных cloud-embedding на 2026 для bge-m3) — задай отдельный
endpoint:

```dotenv
LLM_PROVIDER=cerebras
OPENAI_BASE_URL=https://api.cerebras.ai/v1
OPENAI_API_KEY=csk-…
RAGU_LLM_MODEL=qwen-3-235b-a22b-instruct-2507

EMBEDDINGS_ENABLED=true
EMBEDDING_BASE_URL=http://localhost:11434/v1
EMBEDDING_API_KEY=ollama
RAGU_EMBED_MODEL=bge-m3
```

Если у провайдера нет embedding-endpoint'а — выключи RAG-поиск целиком:

```dotenv
EMBEDDINGS_ENABLED=false
```

Chat при этом работает, а Студия отрисует баннер «загрузка документов
недоступна» (кнопка `+ Источник` дизейблится).

### `mock` для тестов

`LLM_PROVIDER=mock` отдаёт детерминированные ответы по ключевым словам
без реальных запросов наружу — это режим pytest по умолчанию (см.
[`backend/tests/conftest.py`](backend/tests/conftest.py)).

### Размер на M2 Air (рекомендация по железу)

- 8 ГБ RAM — только 3b модель (7b не уместится одновременно с macOS и браузером)
- 16 ГБ RAM — обе модели, переключаться через UI
- 32+ ГБ — можно держать обе одновременно в памяти + большие документы в контексте

## Экспорт в СИГМУ

RAGRAF — это **локальная среда разработки аналитика** в составе фреймворка СИГМА (см. [ТЗ СИГМА §1](docs/TZ_RAGRAF.md)). Аналитик готовит цифровые регламенты тут (импорт PDF/DOCX → извлечение параметров → редактирование в Form/Flow/Constraint), а потом отдаёт готовые артефакты в большую СИГМУ для исполнения.

СИГМА хранит регламенты как Apache Jena граф (RDF/OWL/SHACL). Каждый регламент = пара файлов:
- `data.ttl` — OWL-инстанс с property values
- `shapes.ttl` — SHACL-форма валидации

При загрузке СИГМА автоматически валидирует `data.ttl` против `shapes.ttl` ([Rules-Management.pdf](docs/Rules-Management.pdf), [ТЗ §4.1.3](docs/TZ_RAGRAF.md)).

### Структура bundle

**Один регламент** (`GET /api/regulations/{id}/export-bundle`):

```
<source_id>/
├── data.ttl          # OWL-инстанс: name, date, recommendation, параметры + *Deviation,
│                     # SIGMA-compliance поля (sourceDocument, sourceClause, validFrom/To)
├── shapes.ttl        # SHACL-форма: targetClass Regulation, обязательные поля + типы данных,
│                     # min_inclusive=0 для числовых параметров
└── manifest.json     # RAGRAF-метадата: format_version, exported_at, история, sigma_compliance,
                      # для round-trip импорта обратно в RAGRAF
```

**Весь корпус** (`GET /api/sigma-export/corpus?domain=heating`):

```
ragraf-sigma-corpus.zip
├── pressure-diameter/
│   ├── data.ttl
│   ├── shapes.ttl
│   └── manifest.json
├── heat-inlet-breach/
│   └── ...
├── ...
└── corpus_manifest.json  # сводный manifest со списком included/failed
```

### Что НЕ попадает в экспорт

| Не отдаём | Почему |
|---|---|
| `flow.json` (Rule DSL) | Наш проприетарный формат node-based редактора. СИГМА исполняет регламенты по SHACL + recommendation. Flow остаётся RAGRAF-only. |
| `regulation_history` snapshots | RAGRAF audit-trail. СИГМА ведёт свою историю редакций. Версионирование вынесено в `manifest.json`. |
| Документы аналитика (PDF/DOCX) | Сырые источники, не часть онтологии. |
| RAGU prompt overrides | Локальная конфигурация LLM-стека RAGRAF. |

### Пример data.ttl

```turtle
@prefix : <http://regulations.local/ontology#> .
@prefix owl: <http://www.w3.org/2002/07/owl#> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .

:Regulation a owl:Class .
:pressure a owl:DatatypeProperty ; rdfs:domain :Regulation ; rdfs:range xsd:decimal .
:pressureDeviation a owl:DatatypeProperty ; rdfs:domain :Regulation ; rdfs:range xsd:decimal .

:PressureDiameterRegulation a :Regulation ;
    :name "Регламент на допустимые параметры давления и диаметра..." ;
    :date "2023-10-01"^^xsd:date ;
    :pressure 20.5 ;
    :pressureDeviation 1.5 ;
    :diameter 5.0 ;
    :diameterDeviation 0.2 ;
    :recommendation "Остановите систему, проверьте герметичность..." ;
    :sourceDocument "Постановление № 001/2023" ;
    :sourceClause "1.2" ;
    :validFrom "2023-11-01"^^xsd:date .
```

### Пример shapes.ttl

```turtle
:RegulationShape a sh:NodeShape ;
    sh:targetClass :Regulation ;
    sh:property [ sh:path :name ; sh:datatype xsd:string ; sh:minCount 1 ] ;
    sh:property [ sh:path :pressure ; sh:datatype xsd:decimal ; sh:minCount 1 ; sh:minInclusive 0.0 ] ;
    sh:property [ sh:path :pressureDeviation ; sh:datatype xsd:decimal ; sh:minCount 1 ; sh:minInclusive 0.0 ] ;
    ...
```

SHACL автоматически генерируется под фактический состав параметров регламента — `min_inclusive=0` для всех (нет отрицательных давлений), `*Deviation` парный к каждому параметру.

### Как пользоваться из UI

- **Один регламент**: открой регламент в Редакторе → кнопка **«Экспорт в СИГМУ»** в шапке (рядом с «История»). Скачается `<source_id>-sigma.zip`.
- **Весь корпус**: на странице `/regulations` в toolbar — кнопка **«Экспорт в СИГМУ»**. Скачается `ragraf-sigma-corpus.zip`.

### Документ-основание (PROV-O attachment)

Для traceability «откуда появился цифровой регламент» у каждой записи можно хранить ссылку на оригинал + текст-цитату + локальный кэш файла.

**Зачем.** Аналитик оцифровал бумажный приказ → значения параметров взяты из конкретного предложения текста. Через год заказчик спросит «откуда 20.5 атм?» — можно открыть PDF и показать строку из приказа. Это закрывает требование объяснимости из ТЗ СИГМА §4.2.2 #3.

**Что хранится** (per-regulation):
- `source_url` — внешняя ссылка (Yandex Disk, intranet, любой URL)
- `source_excerpt` — цитата-фрагмент (≤ ~2 КБ)
- `source_file_path` — путь в `<DATA_DIR>/source_documents/{source_id}/<filename>` если файл загружен локально
- `source_checksum` — `sha256:<hex>`, integrity check
- `source_mime_type` — определяется при загрузке

**Лимит размера** локального файла — 25 МБ (PDF-приказы и сканы помещаются). Если оригинал больше — кладём во внешний DMS и сохраняем только URL.

**Сериализация в bundle.** В `data.ttl` через PROV-O (`prov:wasDerivedFrom` → `:Source_*`). СИГМА (Apache Jena) парсит `prov:` нативно:

```turtle
@prefix prov: <http://www.w3.org/ns/prov#> .

:PressureDiameterRegulation
    prov:wasDerivedFrom :Source_PressureDiameter ;
    # … обычные :pressure, :date, etc.

:Source_PressureDiameter a prov:Entity ;
    rdfs:label "Постановление № 001/2023" ;
    :sourceUrl <https://disk.yandex.ru/i/abc123> ;
    :sourceExcerpt "Давление в трубопроводе должно поддерживаться на уровне 20.5 атм…" ;
    :sourceChecksum "sha256:9f2c…" ;
    :sourceMimeType "application/pdf" .
```

Если локальный файл прикреплён — кладётся в ZIP как `<source_id>/source.pdf` (или `.docx` и т.д.) рядом с `data.ttl` / `shapes.ttl`. Manifest получает `source_attachment.file` для self-verification.

**REST:**

| Метод | Путь | Назначение |
|-------|------|-----------|
| `POST` | `/api/regulations/{id}/source-upload` | Multipart файл → сохраняет в `data/source_documents/{id}/`, считает SHA-256, пишет путь/хеш в Regulation |
| `GET` | `/api/regulations/{id}/source-document` | Скачать/preview сам файл (inline для PDF) |
| `DELETE` | `/api/regulations/{id}/source-document` | Удалить только локальный кэш (URL/цитата сохраняются) |
| `GET` | `/api/regulations/{id}/source-verify` | Сверить SHA-256 файла на диске с записанным в БД |

**UI.** На вкладке «Поля» редактора регламента — секция «Документ-основание» (рядом с «Нормативное основание»). Поля: URL, drag-and-drop загрузки файла с показом имени/чексумы, textarea для цитаты, кнопка «Сверить» для re-check'а хеша. На странице списка регламентов — badge «📎 источник прикреплён» если есть локальный файл или URL.

### Импорт SIGMA-bundle обратно в RAGRAF (round-trip)

СИГМА ещё в разработке — пока удобнее править регламенты в RAGRAF и переотдавать. Поэтому работает и обратный путь: если СИГМА (или внешний редактор) изменила bundle, можно загрузить ZIP обратно в RAGRAF и продолжить работу здесь.

**Из UI:**

- **На странице `/regulations`** — рядом с «Экспорт в СИГМУ» кнопка **«Импорт из СИГМЫ»**. Принимает single или corpus ZIP — backend сам различает по структуре. На успех показывается баннер со списком добавленных регламентов.
- **На вкладке «Ограничения»** одного регламента — кнопка **«Импорт SHACL»** принимает `.ttl` ИЛИ `.zip`. Из ZIP'а возьмётся только `shapes.ttl` и применится к текущему регламенту. Если нужно импортировать ВЕСЬ bundle (data.ttl + создание/обновление регламента) — используйте «Импорт из СИГМЫ» на странице списка.

**Через REST:**

```bash
# Импорт SIGMA-bundle (single или corpus)
curl -X POST -F file=@pressure-diameter-sigma.zip \
     http://localhost:8000/api/sigma-import/bundle

# Ответ: { total_imported, total_skipped, total_failed, imported: [...], skipped: [...], failed: [...] }
```

**Что происходит при импорте:**

1. ZIP распаковывается; каждая папка верхнего уровня = один регламент.
2. `source_id` берётся из `manifest.json` (если он от RAGRAF) или из имени папки.
3. `data.ttl` парсится → `Regulation` → сохраняется в DuckDB store как обычная правка с автором `sigma-import`. В историю попадает запись «Импорт SIGMA-bundle».
4. `shapes.ttl` пушится в upstream через `client.update_shapes()`. Если upstream недоступен — мягко пропускается, регламент уже в store. SHACL можно дозагрузить позже через «Импорт SHACL».
5. UI обновляется автоматически (TanStack Query invalidate).

**Гарантия валидации:** у каждого регламента в RAGRAF всегда есть форма `RegulationShape` — либо явно отредактированная во вкладке «Ограничения», либо производная (из параметров: name/date обязательные, recommendation обязателен, каждый параметр decimal ≥ 0 + парный `*Deviation` ≥ 0). Это значит что любой регламент, выгруженный из RAGRAF, СИГМА сможет валидировать (ТЗ §4.1.3 + [Rules-Management.pdf](docs/Rules-Management.pdf)).

### Загрузка в большую СИГМУ

На стороне СИГМЫ (Apache Jena) загрузка bundle'а:
```bash
# Распаковать
unzip pressure-diameter-sigma.zip

# Загрузить в Jena с автоматической валидацией SHACL
# (псевдокод — зависит от CLI вашей сборки СИГМЫ)
jena-load --validate-shacl pressure-diameter/data.ttl pressure-diameter/shapes.ttl
```

`manifest.json` не парсится Jena — это исключительно для трассировки/audit'а на стороне СИГМЫ (откуда пришёл, какой версии, история редакций в RAGRAF).

### Как добавить новый регламент

1. Положи три файла в `backend/data/fixtures/`:
   - `<source_id>.data.ttl` — OWL онтология + инстанс `:Regulation` с парами `<param>` / `<param>Deviation`
   - `<source_id>.shapes.ttl` — SHACL NodeShape с границами и сообщениями `@ru`
   - `<source_id>.flow.json` — стартовый Rule DSL (опционально)
2. Добавь запись в `backend/app/services/fixtures.py:REGISTRY` с `name` и `domain`
3. Если есть новые единицы измерения параметров — допиши в `app/services/turtle_bridge.py:PARAM_UNITS`
4. Если домен новый — добавь его в `fixtures.py:DOMAINS` и в `frontend/src/lib/domains.ts:DOMAIN_VISUALS` (иконка + цвет)
