# Критическая оценка раздела 6 «Описание архитектуры СИГМА» (2025)

> Документ: «Платформа интеллектуального управления городской средой "СИГМА".
> Описание архитектуры. ИИ01-01-02-61-ЛУ», ЦИИНГУ, НГУ, 2025.
> Утвердил: ректор М.П. Фёдорук. Согласовали: А.Н. Люлько, И.А. Болдырев, А.Н. Ряскин.

Цель — оценить § 6 «Диаграмма последовательности движения события» через
призму того, что мы построили в RAGRAF, и предложить корректировки.

> **Поправка после первого обсуждения:** документ СИГМА предполагает
> **локальный LLM** (railway-style деплой на сервере заказчика, не SaaS).
> Это снимает аргумент про токен-затраты на OpenAI. Зато проявляются
> другие узкие места — латентность GPU/CPU, конкуренция за железо
> между Q&A-помощником оператора и маршрутизацией событий, аудит-traceability.
> Также пересмотрены оценки масштаба: реальный сценарий — **локальная
> лицензия + ~100 регламентов на мэрию**, а не 10k. Это меняет акценты,
> но не отменяет архитектурные вопросы.

## TL;DR (поправленный)

Большая часть моих первоначальных опасений про § 6 шаг 6 («CORE → Регламенты/RAG»)
**снижается** на сценарии «100 регламентов + локальный LLM». На таком масштабе
RAG-поиск работает за разумные 50-500 мс, и на потоке 10-100 событий/день
это **не bottleneck** для мэрии.

Однако три архитектурных вопроса остаются актуальными **независимо от масштаба**:

1. **Смешение ролей RAG-помощника и event-router** — это design smell, который
   усложняет наблюдаемость и аудит. Стоит разделить даже на 100 регламентах.
2. **Отсутствие композиционных цепочек между регламентами** — это пропуск
   80% реальных кейсов (МЧС → ЕДДС → ЦУКС, теплосеть → диспетчер → мэр).
3. **Классическая булева логика → false negatives при offline-датчике**.

Также проявляются два не-§6 пробела:
4. Один store на все стадии данных — для аудит-traceability нужно разделение.
5. Аудит-цепочка «event → reg → action → result» описана словами, но не
   формализована как обязательная схема таблиц.

Что у СИГМА **лучше** чем у нас:
- Реестр модулей с паспортом (мы должны добавить).
- API Gateway отдельным компонентом (мы inline в FastAPI).
- Чёткое разделение пилот/тест/прод контуров.

## Корректировка масштаба: как платформа работает в реальности

Прежде чем критиковать, важно зафиксировать **реалистичный жизненный цикл**:

```
СЦЕНАРИЙ «локальная лицензия мэрии»

День 1:   Заказчик покупает лицензию, разворачивает образ платформы
          на собственном сервере (Docker / k8s) в защищённом контуре
          мэрии. Никакого SaaS, никаких внешних API на критическом пути.

Неделя 1-4: Аналитики наполняют базу регламентов:
            - оцифровка существующих приказов (~50-100 регламентов)
            - подключение существующих BI/учётных систем как
              источников событий через ETL-адаптеры
            - настройка webhook-уведомлений на ответственных

Месяц 2+:  Дневной поток событий: 10-1000 событий/день, пики ~50/сек.
           Аналитики иногда уточняют параметры регламентов через UI.
           Оператор СЦ задаёт LLM Q&A-вопросы по корпусу регламентов
           (~5-20 запросов/день).

Год 1+:    Корпус стабилизируется на 100-300 регламентов. Замена
           приходит как версия 2.0 одного регламента, а не сотен новых.
```

В таком сценарии **многие аргументы про масштабирование 10k+ регламентов
не применимы**. RAG-поиск 50 мс × 1000 событий/день = 50 секунд GPU-времени
в сутки — пренебрежимо.

**Но архитектурные принципы важны независимо от масштаба** — потому что:
- они влияют на **наблюдаемость** (можно ли объяснить решение оператора?);
- определяют **расширяемость** (что если завтра подключат шумовой
  мониторинг и поток вырастет до 100k событий/день?);
- задают **семантику аудита** перед прокурором при инциденте.

## Узкое место №1 (пересмотрено): RAG-маршрутизация vs Trigger-index

### Что в СИГМА (§ 6, строки 6-7)

```
6  CORE → Регламенты/RAG:  Запрос применимых цифровых регламентов
7  Регламенты/RAG → CORE:  Возврат правил, порогов, основания проверки
```

На каждом событии CORE спрашивает у RAG-подсистемы «какие регламенты
применимы». RAG — это векторный поиск (embeddings) + опциональный
re-ranking через LLM.

### Поправленная оценка эффекта на 100 регламентах

| Метрика | RAG (local LLM) | Trigger-based индекс |
|---|---|---|
| Latency на событие | 50-300 мс (embedding + ANN) | < 1 мс (B-tree lookup) |
| Throughput при N=100 регламентов | 50-100 событий/сек | > 10 000 событий/сек |
| Стоимость на инстанс | GPU 16-24 ГБ для local LLM | минимальная (CPU + B-tree) |
| Детерминированность | средняя (embedding score threshold) | полная |
| Объяснимость | «similarity 0.78» | «match sensor_subtype = X» |
| Зависимость от железа | сильная — нужен GPU | слабая (CPU достаточно) |
| Корректность при rephrase | нестабильная | стабильная |

На 100 регламентах + 1000 событий/день оба подхода работают. Разница не
в **производительности**, а в **архитектурных свойствах**:

**Объяснимость для прокурора.** При расследовании инцидента нужно ответить:
*«Почему именно этот регламент применился к этому событию?»* Trigger-index
даёт прямой ответ: «`event.sensor_subtype == regulation.trigger.sensor_subtype`».
RAG даёт «similarity 0.82, выбран как top-3 по cosine distance» — это менее
убедительно в суде / в комиссии по расследованию ЧП.

**Конкуренция за GPU.** На локальном инстансе один GPU делит:
- маршрутизацию событий (если она через RAG);
- Q&A-помощника для оператора (если задано);
- генерацию рекомендаций (LLM-обвес если есть).

При пиковом потоке (например, массовое отключение тепла → 500 событий за
2 минуты) RAG-маршрутизация конкурирует с Q&A. Оператор СЦ нажал «спросить
LLM как реагировать» — а ответ ждёт 30 секунд, потому что GPU занят
маршрутизацией бэклога. Trigger-index освобождает GPU только для Q&A.

**Корректность при rephrase регламента.** Аналитик через UI меняет формулировку
регламента: «при превышении 25°С → инцидент» → «при росте температуры выше 25°С
сообщить». Embedding-вектор меняется → similarity ranking тоже меняется → один
event может начать матчиться с **другим** регламентом. С trigger-index это
невозможно: триггер привязан к `sensor_subtype`/`event_type`, формулировка
не влияет.

### Что у нас (RAGRAF)

Декларация триггера живёт в трёх местах **одновременно**, источник правды один:

**1. Turtle data.ttl** — каноническая сериализация для интеграции:
```turtle
:HeatInletBreachRegulation a :Regulation ;
    :hasTrigger :HeatInletBreach_Trigger_trig-inletPressure .

:HeatInletBreach_Trigger_trig-inletPressure a :Trigger ;
    rdfs:label "Давление на входе"@ru ;
    :paramRef "inletPressure" ;
    :sensorSubtype "industrial-pressure" ;
    :eventType "telemetry.pressure" .
```

**2. DuckDB-таблица `regulation_triggers`** с индексами:
```sql
CREATE INDEX idx_trig_subtype ON regulation_triggers(sensor_subtype);
CREATE INDEX idx_trig_event   ON regulation_triggers(event_type);
CREATE INDEX idx_trig_source_reg ON regulation_triggers(source_regulation);
```
[`backend/app/services/regulation_store.py:128-160`](backend/app/services/regulation_store.py)

**3. Flow.json**: `sensor.bindsTo → input.paramRef` — то же самое в графической
модели для редактора.

### Рекомендация (мягкая, под 100 регламентов)

Не «срочно переписывать», а добавить **trigger-index как первичный путь**,
а RAG оставить как fallback для регламентов с нечётко формализованным
триггером:

```
Шаг 6 (новый):
   CORE → Trigger Index: SELECT applicable regulations
   WHERE trigger.sensor_subtype = event.sensor_subtype
      OR trigger.event_type    = event.event_type
   → возвращает 0-N регламентов за < 1 мс
   
Шаг 6a (fallback):
   Если index вернул 0 регламентов И в реестре есть нерасшифрованные
   регламенты без формализованных триггеров — обратиться к RAG для
   семантического поиска.
   → даёт graceful degradation для legacy-регламентов
```

Это **best-of-both-worlds**: новые регламенты сразу формализуются с триггерами
(быстро + объяснимо), старые регламенты постепенно мигрируют, RAG остаётся
как safety net.

## Узкое место №2: композиционные цепочки между регламентами

**(не меняется от масштаба — структурный пробел)**

### Что в СИГМА (§ 6)

Линейный сценарий: «событие → CORE → один регламент → рекомендация». Нет
механизма «output регламента A триггерит регламент B».

### Почему это критично

80% реальных кейсов городского управления — **двух-трёхуровневые**:

```
Звонок 112 → классификация
   ↓ (если массовое)
Перевод ЕДДС в режим повышенной готовности
   ↓ (если региональная ЧС)
Доклад в ЦУКС ГУ МЧС
   ↓ (параллельно при пожаре)
Эскалация в АДПИ-мониторинг
```

Или в коммунальном кейсе:

```
Датчик давления теплосети сработал
   ↓
Регламент «прорыв тепловвода» — рекомендует закрыть smart-valve
   ↓
Регламент «оповещение медблока» — критично если рядом медицинское учреждение
   ↓
Регламент «ЕДДС» — если задействованы службы города
```

В архитектурном документе СИГМА это вообще не описано.

### Что у нас

Поле `RegulationTrigger.source_regulation` указывает: «слушать output другого
регламента, не sensor». Индекс `idx_trig_source_reg` → детерминированный
event-driven каскад.

Реально работает: 4 регламента ЕДДС р.п. Кольцово в цепочке (см.
`backend/data/fixtures/koltsovo-edds-*.flow.json`).

### Рекомендация

Добавить в § 6 диаграмму **шаг 8a**:

```
8a  CORE → ETL: Если вердикт регламента имеет уровень ≥ 1 и output-action,
                инжектировать «событие-вердикт» {source_regulation_id,
                output_action, level} обратно в ETL.
   (Триггерит шаги 5-9 для следующего регламента в цепочке.)
```

Это даёт композицию без переписывания CORE и сохраняет inversion of control:
сам регламент B декларирует «слушаю output регламента A», нет хардкода
цепочек в коде.

## Узкое место №3: смешение RAG-помощника и event-router

**(не меняется от масштаба — design smell)**

### Что в СИГМА

§ 4 определяет одну компоненту «RAG / поиск по регламентам»:

> Поиск применимых правил и документов, поддержка пользователя при анализе
> регламентов и подготовке ответов.

Две разные задачи под одной шапкой:

| Задача | Подходящий механизм | Конкуренция за ресурсы? |
|---|---|---|
| Оператор СЦ задаёт «что делать при разрыве трубы?» | RAG / семантический поиск | Нужен GPU+LLM, ~5-20 запросов/день |
| Event{sensor=industrial-pressure} → applicable regulations | Trigger-based индекс | CPU, ~1000 событий/день |

### Почему это design smell

Они **должны конкурировать за один ресурс** (локальный LLM/GPU). На локальном
инстансе мэрии GPU обычно один. Когда массовый инцидент:
1. 500 событий за 2 минуты — все идут через RAG.
2. Дежурный оператор кричит: «как реагировать на массовое отключение?» —
   нажимает Q&A → его запрос становится в очередь за 500 событиями.
3. Latency Q&A раздувается до минут — оператор работает по памяти, не по
   подсказке системы.

### Что у нас

В RAGRAF мы разделили:
- **RAGU-интеграция** (`backend/app/services/ragu_service.py`) — RAG для Q&A
  по корпусу регламентов. Это поддержка АНАЛИТИКА в Студии аналитика, не
  на критическом пути.
- **`regulation_triggers` + indexes** — для маршрутизации событий. CPU,
  без LLM.

### Рекомендация

Разделить § 4 СИГМА на две компоненты:
- **«Event Router (Trigger Index)»** — между ETL и CORE, на критическом пути
  обработки события. CPU, детерминированный, объяснимый.
- **«RAG / поиск по корпусу для Q&A»** — отдельный сервис, изолирован от
  event-pipeline'а. GPU/LLM, для аналитика и оператора в режиме «спросить
  систему».

Архитектурно это разделение **дешёвое** (просто перенести box в диаграмме),
но избавляет от деградации сервиса в пиковые моменты.

## Узкое место №4: классическая булева логика → false negatives

### Что в СИГМА

§ 6 шаг 8 «CORE → CORE: Оценка критичности, выявление отклонения». Если
sensor offline (event не прилетел), CORE не выявит отклонения и выдаст
«всё ок» — это **false negative**.

### Что у нас

В RAGRAF (фикс 043, см. [README § 8.1 «Опыт математиков»](README.md))
введена трёхзначная логика Клини: result ∈ {true, false, **unknown**}.
При offline-датчике formula возвращает `unknown`, в trace явная пометка
«нет данных: pressure», уровень не выдаётся (level=0 ≠ «всё ок», а
«недостоверно»).

### Рекомендация

В схеме события добавить статус `evidence_level ∈ {measured, derived, unknown}`.
В § 6 шаге 8 правило: «если хотя бы один обязательный параметр события —
unknown, итоговый статус = «недостоверно» с явной пометкой в карточке
руководителю; вердикт уровня НЕ выдаётся».

## Узкое место №5: один store на все стадии

**(не критично на 100 регламентах, но обозначаю как roadmap-вопрос)**

### Что в СИГМА (§ 8)

4 типа данных: исходные / нормализованные / обогащённые события +
регламенты. Но это **логическое** разделение в одной БД.

### Когда это становится проблемой

Не на 100 регламентах и 1000 событий/день. Проблема возникает когда:
- к платформе подключают шумовой / воздушный мониторинг с потоком
  100k+ событий/день;
- read-heavy запросы дашборда руководителя конкурируют с write-heavy ETL;
- аудит-журналы растут быстрее основной таблицы событий → bloat.

### Рекомендация (для roadmap, не на сейчас)

Когда поток событий превысит 50k/день, разделить:
- Сырые/нормализованные события → time-series store (Clickhouse / TimescaleDB)
  с TTL 90 дней.
- Регламенты → реляционный store с MVCC (Postgres / DuckDB).
- Аудит-цепочки → отдельная append-only БД (Postgres logical-replicated в archive).
- Кэш «текущее состояние объектов» → Redis.

В RAGRAF на текущей фазе всё в одной DuckDB — это сознательное упрощение
для прототипа. На production переедем по необходимости.

## Что у СИГМА лучше чем у нас (честно)

1. **Реестр модулей с паспортом** (§ 4 «Реестр модулей»). У нас регламенты
   живут как фикстуры без явного паспорта. Полезно для подключения
   входящих/исходящих сервисов мэрии: каждый модуль декларирует свой
   API-контракт, схему событий, версию.
   → план: добавить поле `module_passport` к каждому regulation_store.

2. **API Gateway как отдельный компонент** (§ 4). У нас FastAPI обрабатывает
   всё в одном app. Для production-инстанса мэрии с rate-limiting и
   API-key auth нужен gateway (Kong / Tyk / nginx).
   → план: на стадии production.

3. **Чёткое разделение пилот / тест / прод контуров** (§ 9). У нас один
   Railway-инстанс. Для лицензируемой платформы это нужно.
   → план: на стадии production.

4. **Аудит-цепочка «событие → регламент → рекомендация → действие пользователя →
   результат»** (§ 2 «Объяснимость»). У нас есть `regulation_history`
   (snapshots регламента), но цепочка действий оператора не записывается.
   → план: добавить таблицу `incident_audit_log` с FK на event_id,
   regulation_id, user_action_id, status.

## Минимальные правки в § 6 СИГМА (поправленные)

### Шаг 6 — расширить, не заменять

```
было: CORE → Регламенты/RAG: Запрос применимых цифровых регламентов
стало: CORE → Trigger Index (индекс по sensor_subtype/event_type):
       O(1) lookup применимых регламентов.
       Если индекс пуст (нет формализованных триггеров для типа события) —
       fallback на RAG-поиск по корпусу для legacy-регламентов.
```

### Шаг 7 — заменить
```
было: Регламенты/RAG → CORE: Возврат правил, порогов, основания проверки
стало: Trigger Index → CORE: Возврат списка [regulation_id].
       CORE подгружает RuleDSL (flow.json + параметры) из контура
       регламентов для каждого regulation_id.
```

### Добавить шаг 8a — композиция
```
CORE → ETL: Если вердикт уровня ≥ 1 и регламент имеет output-action,
            инжектировать «событие-вердикт» {source_regulation_id,
            output_action, level} в ETL → запускается следующий
            регламент в каскаде. (Даёт event-driven эскалацию
            без переписывания CORE.)
```

### Расширить шаг 8 — Kleene unknown
```
было: CORE → CORE: Оценка критичности, выявление отклонения
стало: CORE → CORE: Оценка критичности по трёхзначной логике
       {measured, derived, unknown}. При отсутствии хотя бы одного
       обязательного параметра события — статус «недостоверно»
       с пометкой в карточке руководителю; вердикт уровня НЕ выдаётся.
```

### Дополнить § 4 — разделить RAG-роли
```
RAG / поиск по регламентам → разделить на:
  - Event Router (Trigger Index): между ETL и CORE, CPU, на критическом пути.
  - RAG / поиск по корпусу для Q&A: отдельный сервис, GPU/LLM, изолирован
    от event-pipeline'а. Используется аналитиком и оператором в режиме
    «спросить систему».
```

## Прогноз эффекта (поправленный)

После этих правок СИГМА:
1. На **100-300 регламентах** (типичная мэрия) работает с теми же
   характеристиками + получает объяснимость аудита и стабильность
   роутинга при rephrase регламента.
2. На **1k+ регламентах** (если корпус разрастётся за года эксплуатации)
   сохраняет latency < 5 мс на маршрутизацию вместо 50-500 мс.
3. Закрывает класс ошибок «уверенный false при offline-датчике».
4. Получает event-driven композицию (МЧС → ЕДДС → ЦУКС) без переписывания CORE.
5. Изолирует GPU-нагрузку Q&A от event-pipeline'а → стабильное время
   ответа оператору в пиковые моменты.

Всё это в RAGRAF уже работает и проверено (11 регламентов кампуса НГУ +
4 регламента ЕДДС Кольцово, 0 ошибок валидации, 11/11 e2e тестов). Перенос
архитектурного паттерна — 1-2 человеко-недели на стороне СИГМА (с учётом
того, что они уже спроектировали ETL/CORE/реестр модулей корректно).
