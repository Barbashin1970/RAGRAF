# SKILL-D0SL.md — исследовательский отчёт по D0SL для команды RAGRAF

> **Цель документа.** Систематизировать знания о языке/платформе **D0SL** (разработка компании Eyeline, 2018–2025) на основании двух независимых источников — публичной русскоязычной документации D0SL v2 и локальной копии первой версии SDK `d0SDK-master` — и зафиксировать **прицельные идеи**, которые могут улучшить наш Rule DSL и архитектуру событийных регламентов в RAGRAF / Σ.

> Документ **аналитический**, не операционный. Не описывает как **внедрять** D0SL в RAGRAF; описывает что в нём интересно и где мы уже сильнее.

## 1. Что такое D0SL

### 1.1 Версии и противоречия в источниках

В наших источниках обнаружились **две разные версии** проекта с **разным стеком**:

| Аспект | D0SL v2 (docs, 2024-2025) | D0SL v1 (SDK, 2022) |
|---|---|---|
| Платформа авторинга | **Eclipse-плагин** (репозиторий `updatesite.d0sl.org`) | **JetBrains MPS** 2021.3 (multi-module project) |
| Файлы исходников | Текстовые `.d0sl` файлы | XML-модели MPS (`.mps`) |
| Стандартная библиотека | `org.d0sl.stdlib.DebugDSL`, `JSON` | Аннотации `@DomainModel`, JEP-Python мост |
| Сборка | Не описана в публичной доке | Gradle 4.10 + плагин `mps-gradle-plugin` |
| Активность | 2024-2025, регулярная | Архив, без коммитов после 2022-03 |

Это значит проект **мигрировал** с MPS на Eclipse-tooling и переписал runtime; концепции остались близкими, но конкретный синтаксис и инструментарий **различаются**. В дальнейшем мы для каждого тезиса фиксируем, **из какой версии** он взят.

### 1.2 Расшифровка названия

В **публичной доке D0SL v2 расшифровка отсутствует**. Гипотезы из источников:
- *«Domain-Specific Language»* + цифра 0 в названии — гипотеза по контексту имени (домен `d0sl.org`).
- *«Delta Zero Semantic Language»* — вариант, упомянутый одним из агентов; в README/коде локального SDK подтверждение **не найдено**.
- *«ди-зеро-эс-эл»* — самоназвание по произношению в доке.

Безопасно говорить так: **D0SL — собственное имя**, в исходниках компании однозначной расшифровки не зафиксировано.

### 1.3 Позиционирование (формулировки из доки)

«Семантическая платформа и одноимённый язык для описания логики сложных систем». Целевой пользователь — **domain expert** (предметный специалист), который должен иметь возможность выразить поведение системы без программиста.

С главной страницы D0SL v2 заявлены use cases:
- бизнес-процессы предприятий;
- управление проектами;
- управление автономными системами;
- системы ИИ;
- интернет вещей.

README локального SDK v1 уточняет: язык построен на «теории семантического моделирования», используется для управления роботами; в примерах — **роботакси (Autodrome)** и **алгоритмическая задача 8 ферзей (Chess)**.

## 2. Философия

### 2.1 «Логика против алгоритмов»

Единственный публичный философский тезис (страница `/philosophy/` D0SL v2):

> «Логика позволяет представить знания в более компактной форме. Небольшой набор логических правил позволяет описать очень большое или даже бесконечное множество ситуаций.»

Иллюстрируется аналогией с ПДД: книга правил ~100 страниц охватывает бесконечное множество дорожных ситуаций; алгоритмический перечень «если ситуация X — действие Y» был бы гигантским.

### 2.2 Чего в публичной философии **нет**

Это важно фиксировать, чтобы команда не строила ложные ожидания:

- **Нет** упоминаний Гончарова / Нечесова / Свириденко, SIGMA-методологии или другой формальной школы.
- **Нет** сравнений с BPMN, событийными системами, Drools, OWL/SHACL.
- **Нет** ссылок на теорию моделей, conceptual modeling, формальную семантику.
- **Раздел «Математические основы»** виден в навигации, но страницы `/math/` и `/philosophy/algorithms-vs-logic/` **отдают 404** в версии v2 на момент исследования.

Вывод: D0SL декларирует «логическую» парадигму, но **не публикует** математический фундамент в открытой документации. Это не означает что фундамента нет (он может быть в академических работах команды Eyeline), но в нашем источнике его нет.

## 3. Концептуальная модель

D0SL построен вокруг **трёх фундаментальных сущностей** (D0SL v2, страница `/language/`):

### 3.1 Signature

Объявление **интерфейса**: набор предикатов, функций и пользовательских типов с пометкой `abstract`. Аналог `interface` / header-файла.

```
signature DebugDSL is
    func logStr(a: string) -> logical abstract
end
```

### 3.2 Semantic Model

Основная **единица знаний** на D0SL. Содержит конкретные предикаты, функции и `use`-декларации, импортирующие другие модели. Ключевые слова `model` или `semanticmodel` (взаимозаменяемы в примерах).

```
semanticmodel asdf is
    use debug: DebugDSL from "org.d0sl.stdlib.DebugDSL"
    predicate start() def
        debug.logStr("hello world!")
    end
end
```

### 3.3 Domain Model

**Реализация сигнатуры на внешнем языке программирования** (явно упоминается Java; импортируются по FQCN, например `"org.d0sl.stdlib.DebugDSL"`). Domain Models играют роль драйверов/мостов к внешнему миру: базам данных, CRM, нейросетям, датчикам.

В D0SL v1 (MPS) тот же концепт выражен Java-аннотацией:

```java
@DomainModel(name="AutodromeDSL")
public class AutodromeDSL { ... }
```

### 3.4 Композиция через `use`

```
use <alias>: <Signature> from "<class-or-RMS-address>"
use <alias> @ <Model>
use <alias> with model <Model>
```

В качестве источника модели может выступать **либо Java-класс, либо адрес RMS** (Remote Model Server) — упомянутый в доке протокол для распределённого исполнения моделей. Деталей RMS в публичной доке нет.

### 3.5 Единица исполнения

«**Runnable file** должен содержать модель с предикатом `start()` без аргументов» — интерпретатор стартует с него (D0SL v2).

## 4. Язык: типы и конструкции

### 4.1 Встроенные типы

| Тип | Описание |
|---|---|
| `string` | Текст |
| `num` | Числа с плавающей точкой (внутри Double) |
| `logical` | **Трёхзначная логика Клини**: `true`, `false`, `unknown` |
| `list<type>` | Списки |
| Пользовательские | `type ABC def end` — пустой тип / маркер |

Подчёркнутое — это сильнейшая идея, которой у нас нет. См. §7.

### 4.2 Конструкции

- `model <Name> is ... end` — семантическая модель
- `signature <Name> is ... end` — сигнатура
- `predicate <name>(args) def ... end` — предикат
- `func <name>(args) -> <type> ... return ... end` — функция
- `type <Name> def end` — пользовательский тип
- `var x = ...` — локальная переменная
- `as <type>` / `expect` — приведение типов при доступе к динамическим свойствам

### 4.3 Операторы и выражения

- Логические: `and`, `or`, `!()`, сравнения `==`, `!=`, `<`, `<=`, `>`, `>=`
- **Кванторы**: `forall x in <list>: <logical>` и `exists x in <list>: <logical>`
- Условные: `if-then-else`, тернарный `?(cond ? a : b)`
- Числовые: `+ - * /`
- Строковые: конкатенация через `+`
- Списки: литерал `[a, b, c]`
- Цепочечный доступ: `obj.some.nested.key[0] as num`

### 4.4 Примеры

**Hello world**:

```
model HelloWorld is
    use debug: DebugDSL from "org.d0sl.stdlib.DebugDSL"
    function getGreeting() -> string
        return "Hello world!"
    end
    predicate start() def
        debug.logStr(getGreeting());
    end
end
```

**Квантор всеобщности**:

```
forall x in [1,2,3]: (
    x > 0 and x < 4
)
```

**Динамическая JSON-навигация через domain model**:

```
var obj = j.load("{\"some\": {\"nested\": {\"key\": [1, 2]}}}");
obj.some.nested.key[0] as num == 1;
```

## 5. Tooling и стек

### 5.1 D0SL v2 (текущая публичная версия)

- **Платформа реализации — JVM** (Java-FQCN импорты).
- **IDE — Eclipse**: установка через `Help → Install new software → updatesite.d0sl.org → D0SL Language → Restart`.
- **Файлы** — расширение `.d0sl`.
- **Запуск** — через Eclipse `Run → Run`, вывод в консоль.
- **CLI и автономный компилятор** — в публичной доке **не описаны**. Workflow полностью завязан на IDE.
- **Стандартная библиотека**: явно упомянуты `DebugDSL`, `JSON`. Полный перечень отсутствует.
- **RMS (Remote Model Server)** — упомянутый протокол для распределённых моделей; деталей нет.

### 5.2 D0SL v1 (локальный SDK, архивный)

- **Платформа авторинга — JetBrains MPS 2021.3** (метаязыковая IDE).
- **Сборка — Gradle 4.10** + плагин `mps-gradle-plugin` с GitHub.
- **Runtime — `LocalSemanticMachine`** (Java) в `d0sl_core.jar`:
  - `Calculator` — интерпретатор;
  - `CalculationStack` + `LocalContext` — стек вычисления;
  - `LocalSemanticRegistry` — реестр моделей и библиотек;
  - `SemanticEnvironment` — контекст выполнения.
- **Python-мост через JEP** (Java Embedded Python 3.8.2). Файл `someDsl.py` содержит эмулятор, преобразующий вызовы D0SL в методы Python.
- **Примеры**:
  - `Autodrome` (управление роботакси) — `auto.drome.mps` ~7 400 строк XML;
  - `Chess` (задача о 8 ферзях) — `chess.mps` ~2 000 строк XML.
- **Зрелость**: snapshot без git-истории; 74 файла, ~7.4 MB; **тестов нет**; документация — README на 11 строк + javadoc в коде.

### 5.3 Где D0SL хранит знания

В обеих версиях знания хранятся **в коде на D0SL**, не во внешнем триплсторе. Отсутствуют интеграции:
- RDF / Triple Store — нет;
- OWL — нет;
- SHACL — нет;
- Prolog-style reasoner — нет.

Логический слой реализован **классами AST** (Logical, ForAll, Implication, Comparison, And/Or/Not) поверх собственной FOL-семантики.

## 6. Сильные стороны и ограничения

### 6.1 Сильные

- **Декларативный язык с кванторами и трёхзначной логикой** → компактное выражение знаний, читаемое предметным специалистом.
- **Чёткое разделение Signature / Semantic Model / Domain Model** — интерфейс/реализация/адаптер. Даёт переносимость и подмену реализации.
- **RMS** — концепция распределённого исполнения моделей (на бумаге).
- **Прямой `forall` / `exists`** над коллекциями — то, чего не хватает многим Rule-DSL.
- **Унифицированный AST** (Predicate / Function / Variable / Expression) с понятной иерархией операторов — крепкая база для интерпретации и потенциальной верификации.

### 6.2 Ограничения

- Публичная документация **очень тонкая**. Философия — один тезис. Математические основы — 404. Семантика языка нигде не оформлена как стандарт.
- **Tooling привязан к IDE** (Eclipse в v2, MPS в v1); нет CLI, REPL, Web-IDE в доступной доке.
- **Domain models = Java** → порог входа выше декларируемого «понятный domain-специалисту».
- **Нет онтологического слоя**: ни RDF, ни OWL, ни SHACL, ни автоматического reasoner'а. Знания живут только в файлах языка.
- **Нет событийной модели**: триггеры, темпоральная логика, реактивность — не упомянуты в доке.
- **Нет визуального редактора** ни в v2, ни в v1 (MPS даёт проекционный редактор, но это редактор кода, не графа процесса).
- **Проект v1 заброшен с 2022**; v2 публикуется, но без явного roadmap.

## 7. Сравнение D0SL ↔ RAGRAF Rule DSL

| Аспект | D0SL | RAGRAF Rule DSL |
|---|---|---|
| **Парадигма** | Логические предикаты + кванторы + функции (FOL-подобная) | Граф узлов flow (Node-RED-like), 7 типов узлов |
| **Редактор** | Текстовый код в IDE (Eclipse / MPS) | Визуальный React Flow с drag-and-drop |
| **Логика истинности** | **Трёхзначная (Клини): true/false/unknown** | Булевская (true/false) |
| **Кванторы** | `forall` / `exists` над списками | Нет; «все датчики OK» моделируется каскадом нод |
| **Типизация** | string/num/logical/list + пользовательские типы из сигнатур | decimal/string/date/boolean; нет пользовательских типов |
| **Композиция** | `use <alias>: Signature from <impl>` | Триггеры `source_regulation`/`source_output` + `:hasTrigger` в Turtle |
| **Источник правды** | Текст `.d0sl` (или MPS-XML) | DuckDB store + Turtle (SIGMA-bundle) |
| **Онтология / семантика** | Нет | OWL/Turtle через rdflib + SHACL валидация |
| **Reasoner** | Собственный интерпретатор `LocalSemanticMachine` | `flow_executor` + опциональный RAGU (GraphRAG) |
| **Событийная сцепка** | Нет в публичной модели; стартовая точка — `start()` предикат | `RegulationTrigger` с sensor_subtype / event_type / source_regulation |
| **Внешний мир** | Java domain models по FQCN | Sensor Library с подтипами + JSON payload-схемами + Sensor-нода в flow |
| **Визуализация** | Нет | Cytoscape Graph View + React Flow Editor |
| **Темпоральные операторы** | Нет в доке | Нет (есть `confirmTimeMinutes`, `escalationTimeMinutes` как параметры — не как операторы) |
| **Распределённость** | RMS (бумажный концепт) | HTTP-API + SIGMA-bundle round-trip |

Главный вывод: **D0SL и RAGRAF — это разные классы инструментов**, у которых пересечение в одном слое (декларация правил), но всё остальное — про разное. D0SL — это **язык программирования с логической парадигмой**. RAGRAF — это **визуальная авторинг-система для регламентов**, привязанная к индустриальной онтологии (SHACL/OWL) и event-driven рантайму. Они не конкуренты; они для разных пользователей и разных задач.

## 8. Идеи для заимствования

Прицельный список того, что **технически осуществимо** взять в RAGRAF без перестройки архитектуры.

### 8.1 🟢 Трёхзначная логика для `compare`/`threshold` (sigma:medium)

**Сейчас:** узел `compare` возвращает `true`/`false`. Если sensor offline, value = NaN — flow_executor неявно даёт `false`, что семантически некорректно.

**Заимствование:** ввести **третье значение `unknown`** для:
- отсутствующих SensorReading на момент исполнения;
- значений вне домена (NaN, тип не сматчился);
- параметров с datatype=`unknown` (например, текст без числового значения).

**Что меняется:**
- Поле `compare.result: 'true' | 'false' | 'unknown'`;
- Output-нода уровня `level=0` (норма), `level=N` (срабатывание), новое: `level=-1` (недостоверно — не выдаём вердикт);
- В trace explicit-ная пометка `unknown: sensor offline`.

**Выгода:** честные отчёты в Execute Layer, защита от «уверенного false» когда данных нет. Полезно для интеграции с SHACL — там `sh:NodeShape` уже различает `Violation` vs. `Warning` vs. unknown conformance.

### 8.2 🟢 Кванторы `forall` / `exists` как новый тип узла (sigma:medium)

**Сейчас:** «все датчики комнаты в норме» = N input/threshold/compare-нод соединённых через `formula(a && b && c)`. Невыразимо для динамической группы датчиков.

**Заимствование:** новый узел `quantifier`:
- `kind: 'forall' | 'exists'`
- `scope: <выражение, возвращающее list>` (например `zone:server-room.sensors`)
- `predicate: <ссылка на под-flow или formula-выражение>`
- семантически: «для всех элементов scope выполняется predicate»

**Выгода:** одна нода вместо каскадов; работает с динамическим scope (новый датчик в зоне автоматически попадает в проверку).

**Сложность:** требует ввести в Rule DSL понятие **итерируемой коллекции** на runtime — сейчас input-ноды атомарны.

### 8.3 🟡 Signature/Implementation для регламента (sigma:high)

**Сейчас:** регламент — монолит: параметры + flow + триггеры. Подменить «реализацию» (например, для unit-теста или альтернативной логики) нельзя.

**Заимствование:** ввести **`RegulationSignature`** как отдельную сущность:
- объявляет: какие входные параметры (по типам), какие output-actions, какие триггеры (контракт);
- **`Regulation`** реализует одну `Signature`;
- одна Signature может иметь N реализаций (production / fallback / test mock).

**Выгода:**
- SHACL-валидация **уровня контракта**: «реализация соответствует обещанной сигнатуре»;
- A/B-сценарии: две версии регламента на одной сигнатуре;
- Переиспользование под-регламентов как библиотек.

**Сложность:** меняется модель данных, требуются миграции в DuckDB. Целесообразно когда у пользователей появятся первые «семейства» регламентов.

### 8.4 🟢 Пользовательский тип параметра помимо decimal/string/date/boolean (sigma:low)

**Сейчас:** `Parameter.datatype` — `decimal | string | date | boolean`. Это узко: нельзя выразить `coordinates`, `licensePlate`, `incidentLevel` как именованный тип.

**Заимствование:** идея D0SL `type ABC def end` — **именованные типы-маркеры**. У нас в Sensor Library уже есть `SensorSubtype.field.datatype` — можно расширить тот же словарь и для `Parameter.datatype`, добавив пользовательские типы через DuckDB-таблицу `parameter_types` (id, label, base_datatype, validation_regex).

**Выгода:** валидация параметра «номер ГРЗ» по regex автоматически, без хардкода в коде.

### 8.5 🟡 Унифицированное Expression-выражение в `formula`-ноде (sigma:medium)

**Сейчас:** узел `formula.expression` — строка, **не парсится** (MVP). Это огромный пробел: формула может быть только мнемоническим описанием.

**Заимствование:** взять у D0SL парсер выражений (FOL-операторы + кванторы + цепочечный доступ). Это минимальный самостоятельный язык внутри узла flow:
```
formula: forall sensor in zone.sensors: sensor.value < 50.0
```
Альтернативно — взять уже готовое (Python `simpleeval`, Java `MVEL`, JS `expr-eval`) и обернуть в безопасный sandbox.

**Выгода:** правила, которые сейчас требуют 5 узлов, выражаются одной строкой. Декларативные ограничения на длину выражения и dotted-paths сохранят понятность.

**Сложность:** парсинг + безопасный eval; обеспечение детерминизма.

### 8.6 🟡 Объяснимость через trace-цитирование правила (sigma:low)

**Сейчас:** `ExecutionResult.trace` содержит человеко-читаемые объяснения, но они вырабатываются flow_executor'ом «руками». Нет машинно-читаемой связи с самим правилом.

**Заимствование у D0SL:** идея, что **каждое срабатывание имеет цитату из predicate'а**, который привёл к решению. У нас можно прокидывать в trace **выражение узла дословно** + значения переменных в момент срабатывания. Это уже частично есть в Execute Layer, но не унифицированно.

## 9. Чего НЕ стоит брать

- **Привязку к IDE.** Eclipse в v2 и MPS в v1 — антипаттерн для широкого внедрения. Наш React Flow в браузере — лучшее UX.
- **Java-FQCN-импорты Domain Models.** Это лишний слой связи с конкретным runtime. У нас Sensor Library и REST-эндпоинты — более развитый паттерн.
- **Отсутствие онтологического слоя.** OWL/SHACL у нас — преимущество, отказ от него ради «простоты языка» — регресс.
- **Отсутствие версионирования и истории.** D0SL хранит код в файлах; у нас DuckDB-history со snapshot'ами на каждый save — это требование §4.1.3 ТЗ Σ и его терять нельзя.
- **Закрытость экосистемы.** Standard library D0SL непублична; стандарт языка не оформлен. Это барьер для интеграции.

## 10. Открытые вопросы

Что хотелось бы выяснить, прежде чем серьёзно заимствовать:

1. **Математические основы D0SL.** В навигации `/docs/ru/` есть раздел, но страницы 404. Если он связан с школой Гончарова/Свириденко (SIGMA-методология) — это сильное пересечение с нашим теоретическим бэкграундом и стоит копать. Если просто FOL — то наше пересечение слабее, чем казалось.
2. **RMS protocol.** Концепт распределённого исполнения моделей — упомянут, не специфицирован. Может пересекаться с нашим SIGMA-bundle round-trip и/или с будущим ETL-приёмником.
3. **Темпоральные операторы.** В индустриальных регламентах нужны конструкции вида «давление выше нормы более 5 минут». Ни в D0SL v1, ни в v2 такого в публичных источниках не нашли — но это может быть в академических работах Eyeline.
4. **Верификация / model checking.** Заявленная «логичность» предполагает доказательство свойств модели. В коде SDK v1 этого слоя нет. Не путать с runtime-валидацией.
5. **Кто реальный пользователь.** В доке заявлен «domain expert», но Eclipse/MPS + Java-импорты противоречат этому позиционированию. Кто на самом деле пишет на D0SL и для каких задач — не задокументировано.

---

## 11. Резюме одной фразой

D0SL — **компактный декларативный язык с FOL-семантикой и кванторами**, разработанный как пользовательский DSL для domain experts, но **в публичной форме недоразвитый**: документация фрагментарная, IDE-only tooling, нет онтологического слоя и событийной модели. У него есть **3-4 конкретные идеи** (трёхзначная логика, `forall`/`exists`, signature/implementation, парсер выражений), которые мы можем точечно перенести в RAGRAF, не меняя архитектуру. Сам D0SL **не заменяет** наш Rule DSL — он работает в другом классе задач.
