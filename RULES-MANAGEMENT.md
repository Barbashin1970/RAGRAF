# Управление регламентами в Сигма (с триггерами)

> **Пояснительная записка команде.** Документ переписывает исходный «Rules-Management.pdf» (стартовое ТЗ Сигмы по управлению регламентами) с учётом того, что в RAGRAF мы дополнили модель **триггерами** — декларативной привязкой «событие → регламент». Здесь объясняется зачем, как и какие следствия для архитектуры платформы.

---

## 1. Что остаётся неизменным (база Сигмы)

Для хранения и управления регламентами Сигма использует **Apache Jena** — открытый фреймворк для работы с данными в формате **RDF/OWL**. Регламент — это формализованное описание сущностей бизнес-логики, метаданных и правил, описанное по стандарту **SHACL (Shapes Constraint Language)**.

**Порядок работы остался прежним:**

1. **Определение форм SHACL** — анализ существующих текстовых регламентов, выделение основных форм объектов, формулировка требований и ограничений к наполнению.
2. **Описание регламентов в OWL/Turtle** — человекочитаемый формат, редактируется через UI, валидируется против SHACL.
3. **Исполнение регламента** — ETL принимает данные с внешних источников, на их основе выбирается подходящий регламент, оценивается критичность отклонения, формируется рекомендация.

Эта триада (формы → инстансы → исполнение) — фундамент, и мы его не меняем.

---

## 2. Где у исходной модели прячется проблема

Возьмём оригинальный пример — регламент про давление и диаметр трубопровода:

```turtle
:PressureAndDiameterRegulation
    a                  :Regulation ;
    :name              "Регламент на допустимые параметры давления и диаметра ..." ;
    :date              "2023-10-01"^^xsd:date ;
    :pressure          20.5 ;
    :pressureDeviation 1.5 ;
    :diameter          5.0 ;
    :diameterDeviation 0.2 ;
    :recommendation    "Остановите систему, проверьте герметичность..." .
```

И сценарий исполнения из ТЗ — ETL приносит данные от датчика:

```json
{ "edge_id": 1, "value": 0.087, "reference": 0.1, "type": "p", "level": 3 }
```

Дальше выполняется SPARQL-запрос, который находит регламент по `type: "p"` → возвращает `level + regulation + recommendation`.

**На одном параметре всё хорошо. Проблемы начинаются, когда:**

### 2.1 Регламент имеет несколько входов

Реальные регламенты редко зависят от одного параметра. Пример из нашего корпуса — `heat-inlet-breach` (прорыв на тепловом вводе): чтобы система решила, что произошёл прорыв, ей нужно сопоставить **три независимых сигнала**:

| Параметр              | Тип события            | Источник                  |
| --------------------- | ---------------------- | ------------------------- |
| `inletPressure`       | `telemetry.pressure`   | манометр на входе         |
| `pressureFallRate`    | `telemetry.pressure`   | производная того же канала|
| `inletTemperature`    | `telemetry.temperature`| RTD-термопара             |

Поле `type: "p"` в payload события однозначно не отвечает на вопрос «какой именно регламент это активирует». Один и тот же тип `p` может прийти и в `heat-inlet-breach`, и в `pressure-diameter`, и в чём-то третьем. Нужна более тонкая привязка — на уровне **конкретного параметра** регламента, а не его типа.

### 2.2 Связь «датчик → регламент» нигде не объявлена

В исходной схеме регламент знает свои `pressure`/`diameter`/`recommendation` — но не знает, **какие датчики и какие типы событий его приводят в действие**. Этот knowledge живёт за пределами регламента, в коде ETL-роутера. Если завтра добавить новый регламент или новый тип датчика, ETL-роутер придётся менять руками — а это противоречит идее «регламент описывает себя сам».

### 2.3 SPARQL-маршрутизация по типу = O(N) от размера корпуса

Когда в Сигме будет не 11 регламентов, а 10 000 — SPARQL-запрос «найди регламент с `:pressure`» вернёт сотни кандидатов. Дальше всё равно надо как-то выбирать «правильный». Без явной привязки **датчик → параметр конкретного регламента** это вырождается либо в LLM-классификатор (медленно и недетерминированно), либо в hand-coded маршрутизацию (то самое, чего мы хотели избежать).

## 3. Библиотека JSON-контрактов событий (Sensor Library)

Прежде чем регламент сможет «подписаться на событие от датчика», нужно знать, **что вообще шлёт этот датчик** — какой у события набор полей, какие типы данных, какие единицы. Исходное ТЗ Сигмы этот слой не описывало явно: предполагалось, что payload известен заранее и зашит в код ETL. Это нормально для прототипа на одном-двух датчиках; на проде с десятками типов источников событий это становится тонким местом.

В RAGRAF мы вынесли это в отдельную **редактируемую библиотеку JSON-контрактов** — маршрут `/sensors` в UI, бэкенд `services/sensor_schema_store.py`. Цель — дать аналитику первоклассный реестр «всех типов событий, которые могут прийти в систему», не требуя правок кода при добавлении нового датчика.

### 3.1 Двухуровневая модель: класс → подтип → поля

| Уровень    | Назначение                                                | Где живёт              | Кол-во в seed |
| ---------- | --------------------------------------------------------- | ---------------------- | ------------- |
| **Класс**  | Базовое семейство датчиков (физическая природа сигнала). Литерал `SensorType` в коде. | `app/schemas/domain.py` | **7** |
| **Подтип** | Конкретная модель внутри класса (продукт / алгоритм). Динамически добавляется из UI. | DuckDB `sensor_subtypes` | **22** |
| **Поле payload** | Конкретное поле JSON-события подтипа (имя, тип, единица). | DuckDB `sensor_field_schemas` | **282** |

**7 классов:**
- `p` — давление (манометр, преобразователь давления)
- `t` — температура (термопара, RTD)
- `flow` — расход (электромагнитный/ультразвуковой расходомер)
- `noise` — точечный акустический (детектор утечки)
- `detector` — видеодетектор (CCTV-аналитика)
- `fiber` — распределённое оптоволокно (DAS, Distributed Acoustic Sensing)
- `air` — качество воздуха (CO2, PM2.5, PM10, NO2)

**22 подтипа в seed-наборе** — это не выдуманные примеры, а реальные продакшн-каталоги:
- **10 видеодетекторов Нетрис** (по PDF 2024): ANPR (`vd-anpr`), детекция людей (`vd-person`), мусорные баки (`vd-trash-bin`), детекция лица (`vd-face`), и т.д.
- **6 транспортных детекторов Войслинк** (`vl-*`)
- **2 DAS-подтипа**: `fiber-vibration`, `fiber-acoustic`
- **3 air-варианта**: `air-co2`, `air-pm`, `air-no2`
- **7 generic-per-class** (по одному на каждый класс — точки расширения)

### 3.2 Структура хранения

```sql
CREATE TABLE sensor_subtypes (
    subtype_id   VARCHAR PRIMARY KEY,   -- 'vd-anpr', 'fiber-vibration', ...
    class_id     VARCHAR NOT NULL,      -- один из 7 литералов SensorType
    label        VARCHAR NOT NULL,      -- 'ANPR (распознавание ГРЗ)'
    description  VARCHAR,
    position     INTEGER DEFAULT 0
);

CREATE TABLE sensor_field_schemas (
    subtype_id     VARCHAR NOT NULL,     -- FK на sensor_subtypes
    field_name     VARCHAR NOT NULL,     -- 'numberPlate', 'pm25', 'pressure', ...
    datatype       VARCHAR NOT NULL,     -- 'decimal' | 'integer' | 'string' | 'boolean'
    unit           VARCHAR,              -- 'atm' | '°C' | 'µg/m³' | NULL
    description    VARCHAR,
    required       BOOLEAN,
    example_value  VARCHAR,              -- JSON-encoded example
    position       INTEGER,
    PRIMARY KEY (subtype_id, field_name)
);
```

Поля редактируются из UI через REST-эндпоинты (`PUT /api/sensor-schemas/{subtype_id}/{field_name}`, `DELETE`, и т.д.). CRUD на любом уровне — класс зафиксирован в коде, но подтипы и поля свободно создаются/правятся/удаляются.

### 3.3 Пример: контракт `vd-anpr` (видеодетектор ГРЗ)

**Поля подтипа** (выборка из 18 полей в seed):

| field_name      | datatype | unit | required | example_value                        |
| --------------- | -------- | ---- | -------- | ------------------------------------ |
| `numberPlate`   | string   | —    | ✓        | `"А123БВ77"`                         |
| `confidence`    | decimal  | —    | ✓        | `0.94`                               |
| `country`       | string   | —    |          | `"RU"`                               |
| `lpType`        | string   | —    |          | `"car"`                              |
| `direction`     | string   | —    |          | `"in"`                               |
| `cameraId`      | string   | —    | ✓        | `"camera-01"`                        |
| `bbox`          | string   | —    |          | `"{\"x\":120,\"y\":340,\"w\":210,\"h\":48}"` |
| `vehicleColor`  | string   | —    |          | `"silver"`                           |
| ...             |          |      |          |                                      |

**JSON-skeleton события** (так UI показывает превью под таблицей полей):

```json
{
  "description": "<событие от vd-anpr>",
  "timestamp": "2026-05-17T08:42:11Z",
  "payload": {
    "numberPlate": "А123БВ77",
    "confidence": 0.94,
    "country": "RU",
    "lpType": "car",
    "direction": "in",
    "cameraId": "camera-01",
    "bbox": "{\"x\":120,\"y\":340,\"w\":210,\"h\":48}",
    "vehicleColor": "silver"
  }
}
```

Это и есть **JSON-контракт события** — то, что Сигма ожидает увидеть на шине от любого датчика с `subtype_id = "vd-anpr"`. Reproducible, editable, версионируемый.

### 3.4 Источник правды для критичных подтипов — ORM продакшн-систем

Для двух базовых подтипов (`vd-person`, `vd-anpr`) seed-данные приходят **не из выдумки, а из реальных ORM-файлов** продакшн-видеодетекторов:

```
event-data-examples/videodetectors/person_postgresql.py   →  поля для vd-person
event-data-examples/videodetectors/grz_postgresql.py      →  поля для vd-anpr
```

Это значит контракт `vd-anpr` в RAGRAF — это **бит-в-бит схема продакшн БД** видеодетектора Нетрис. При импорте сидера читается Python ORM, извлекаются `Column` declarations, мапятся на наши `field_name + datatype + unit`. Если завтра вендор добавит поле — оно появится в seed после re-import. Хардкода и угадывания нет.

### 3.5 Из библиотеки в Rule DSL Flow Editor

JSON-схемы датчиков напрямую используются в **редакторе регламентов**:

1. В Flow Editor добавляем `sensor`-ноду (оранжевый кружок) → PropertyPanel показывает селект «класс + подтип».
2. После выбора подтипа PropertyPanel рендерит **JSON-preview payload'а** прямо из реестра — аналитик видит, какие поля доступны для условий.
3. Ребро `sensor → input` автоматически проставляет `bindsTo` (связь sensor-узла с input-узлом регламента).

То есть аналитик в реальном времени работает с тем же контрактом, который будет получен из шины СИГМЫ. Никакой угадайки «а что приходит от этого датчика».

### 3.6 Из библиотеки в триггеры (следующий раздел)

И главное: именно `subtype_id` подтипа — это та самая identity, которую несёт триггер регламента (поле `sensor_subtype` в `RegulationTrigger`). Когда регламент декларирует «я слушаю датчик `industrial-pressure`», эта строка указывает на конкретную запись в `sensor_subtypes` — со всеми её полями, единицами, JSON-skeleton'ом. Триггер не висит в воздухе — он привязан к реальному, редактируемому, документированному JSON-контракту.

Это и есть **первая опора сквозной правды**: датчик описан → регламент на него подписан → событие приходит в этом контракте → триггер находит регламент. Без потери identity на любом из переходов.

REST-эндпоинты библиотеки (полный список):

- `GET / POST /api/sensor-subtypes` — список / создание подтипа
- `PUT / DELETE /api/sensor-subtypes/{id}` — обновление / удаление
- `GET /api/sensor-subtypes/{id}/regulations` — **reverse-lookup**: какие регламенты слушают этот подтип (через триггеры)
- `GET /api/sensor-subtypes/_usage` — агрегат `subtype_id → N регламентов` для бэйджей
- `GET /api/sensor-schemas` — все subtype_id → массив полей
- `GET /api/sensor-schemas/{subtype_id}` — поля одного подтипа
- `PUT / DELETE /api/sensor-schemas/{subtype_id}/{field_name}` — upsert / удалить поле
- `POST /api/sensor-schemas/reseed` — сбросить и пересеять дефолтный набор (на dev-машинах)

## 4. Что мы докрутили: триггеры как первоклассная сущность регламента

В RAGRAF регламент дополнен новым полем — **`triggers`** (список `RegulationTrigger`). Это **декларативная привязка** «вход регламента ↔ источник события». Хранится:
- **в Turtle** как объекты класса `:Trigger`, связанные с регламентом через предикат `:hasTrigger` — для онтологической чистоты и SIGMA-bundle экспорта;
- **в DuckDB-таблице** `regulation_triggers` с индексами на `sensor_subtype` и `event_type` — для O(1) маршрутизации на исполнении.

### 4.1 Расширенный OWL-пример (тот же регламент, теперь с триггерами)

```turtle
@prefix : <http://regulations.local/ontology#> .
@prefix owl: <http://www.w3.org/2002/07/owl#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .

# Базовые декларации (как было в исходном ТЗ):
:Regulation        a owl:Class .
:pressure          a owl:DatatypeProperty ; rdfs:range xsd:decimal .
:pressureDeviation a owl:DatatypeProperty ; rdfs:range xsd:decimal .
:diameter          a owl:DatatypeProperty ; rdfs:range xsd:decimal .
:diameterDeviation a owl:DatatypeProperty ; rdfs:range xsd:decimal .

# Новые декларации (RAGRAF):
:Trigger           a owl:Class .
:hasTrigger        a owl:ObjectProperty ;
                   rdfs:domain :Regulation ;
                   rdfs:range  :Trigger .
:paramRef          a owl:DatatypeProperty ; rdfs:range xsd:string .
:sensorSubtype     a owl:DatatypeProperty ; rdfs:range xsd:string .
:eventType         a owl:DatatypeProperty ; rdfs:range xsd:string .

# Инстанс регламента:
:PressureAndDiameterRegulation
    a                  :Regulation ;
    :name              "Регламент на допустимые параметры давления и диаметра ..." ;
    :date              "2023-10-01"^^xsd:date ;
    :pressure          20.5 ;
    :pressureDeviation 1.5 ;
    :diameter          5.0 ;
    :diameterDeviation 0.2 ;
    :recommendation    "Остановите систему, проверьте герметичность..." ;
    :hasTrigger        :PressureAndDiameter_Trigger_trig-pressure .

# Триггер — отдельный субъект:
:PressureAndDiameter_Trigger_trig-pressure
    a                  :Trigger ;
    rdfs:label         "Давление в трубопроводе"@ru ;
    :paramRef          "pressure" ;
    :sensorSubtype     "industrial-pressure" ;
    :eventType         "telemetry.pressure" .
```

Триггер — **отдельный субъект** в графе, а не вложенная структура внутри регламента. Это решает три задачи:
1. SPARQL может фильтровать триггеры независимо от регламентов (`SELECT ?reg WHERE { ?reg :hasTrigger/:sensorSubtype "industrial-pressure" }`).
2. Один датчик может «слушаться» из нескольких регламентов — тогда у нас N триггеров с одинаковым `:sensorSubtype`, привязанных к разным регламентам.
3. Триггеры версионируются и валидируются SHACL отдельно (минимальный SHACL-shape ниже).

### 4.2 SHACL-форма для триггера

```turtle
:TriggerShape a sh:NodeShape ;
    sh:targetClass :Trigger ;
    sh:property [ sh:path :paramRef ;      sh:datatype xsd:string ; sh:minCount 1 ] ;
    sh:property [ sh:path :sensorSubtype ; sh:datatype xsd:string ; sh:minCount 0 ] ;
    sh:property [ sh:path :eventType ;     sh:datatype xsd:string ; sh:minCount 0 ] .
```

`paramRef` обязателен — это FK на имя параметра регламента (`:pressure`, `:inletPressure`, …). `sensorSubtype` и `eventType` опциональны — триггер может быть «незавершённым» (аналитик ещё не привязал датчик).

### 4.3 Множественность входов — то, ради чего всё затевалось

Регламент `heat-inlet-breach` в нашей реализации несёт **5 триггеров**:

```turtle
:HeatInletBreachRegulation
    a            :Regulation ;
    :name        "Прорыв на тепловом вводе" ;
    :inletPressure       4.0 ;   :inletPressureDeviation       0.5 ;
    :pressureFallRate    0.0 ;   :pressureFallRateDeviation    0.2 ;
    :inletTemperature   70.0 ;   :inletTemperatureDeviation   10.0 ;
    :walkerConfirmTime   7.0 ;   :walkerConfirmTimeDeviation   3.0 ;
    :medicalAlertWindow 15.0 ;   :medicalAlertWindowDeviation  5.0 ;
    :hasTrigger
        :HeatInletBreach_Trigger_trig-inletPressure,
        :HeatInletBreach_Trigger_trig-pressureFallRate,
        :HeatInletBreach_Trigger_trig-inletTemperature,
        :HeatInletBreach_Trigger_trig-walkerConfirmTime,
        :HeatInletBreach_Trigger_trig-medicalAlertWindow .

:HeatInletBreach_Trigger_trig-inletPressure
    a :Trigger ;
    :paramRef       "inletPressure" ;
    :sensorSubtype  "industrial-pressure" ;
    :eventType      "telemetry.pressure" .
# ... и так далее для остальных 4
```

Каждый триггер несёт «свой канал» — свой датчик, свой тип события, свой параметр. Один регламент = N независимых сигналов, каждый со своим контрактом.

## 5. Как меняется поток исполнения

### 5.1 Было (по исходному ТЗ)

```
[датчик] → ETL → { type: "p", value: 0.087, reference: 0.1 }
                     ↓
           [SPARQL по типу: SELECT ?reg WHERE { ?reg :pressure ?_ }]
                     ↓
           кандидаты-регламенты (1..N)
                     ↓
           [хардкод-логика: выбрать "подходящий"]
                     ↓
           { level: 3, regulation: "...", recommendation: "..." }
```

Слабое звено — middle layer «выбрать подходящий». Без явной декларации привязки этот выбор либо хардкодится в ETL, либо угадывается LLM.

### 5.2 Стало (с триггерами)

```
[датчик industrial-pressure] → ETL → {
    sensor_subtype: "industrial-pressure",
    event_type:     "telemetry.pressure",
    value: 0.087,
    edge_id: 1
}
    ↓
[SQL по индексу regulation_triggers]:
    SELECT regulation_id, trigger_id, param_ref
    FROM   regulation_triggers
    WHERE  sensor_subtype = 'industrial-pressure'
       OR  event_type     = 'telemetry.pressure'
    ↓
[N точно адресованных триггеров → N (regulation, param_ref) пар]
    ↓
[для каждого: flow_executor.execute_flow(regulation, {param_ref: value})]
    ↓
[вердикты { level, recommendation } по каждому сработавшему регламенту]
```

Изменения принципиальные:
- **Маршрутизация O(1) от индекса**, а не O(N) от обхода всех регламентов.
- **Декларация полная**: ETL вообще не нужно знать о топологии регламентов — он спрашивает базу «кому это интересно?».
- **Привязка к конкретному `param_ref`** (а не просто к «регламенту вообще») — flow_executor сразу знает, в какую input-ноду подставить значение.
- **Эквивалентный SPARQL-запрос** работает на той же базе (для систем без DuckDB):
  ```sparql
  SELECT ?reg ?param_ref WHERE {
      ?trig a :Trigger ;
            :sensorSubtype "industrial-pressure" ;
            :paramRef ?param_ref .
      ?reg :hasTrigger ?trig .
  }
  ```

## 6. Почему дублируем хранение (Turtle + DuckDB)

На первый взгляд — нарушение DRY: одни и те же триплы лежат и в OWL-графе Apache Jena, и в реляционной таблице. На практике каждый слой решает свою задачу:

| Слой                 | Назначение                                                                 | Цена доступа       |
| -------------------- | -------------------------------------------------------------------------- | ------------------ |
| Turtle / RDF / OWL   | Источник правды. Экспортируется в SIGMA-bundle. Версионируется. Валидируется SHACL. | Парсинг ~единиц мс на регламент |
| DuckDB `regulation_triggers` | Производный индекс. Пересоздаётся при сохранении регламента. Используется ETL-приёмником. | <1мс на 10⁴ строк |

Это типовой read-model паттерн: write goes to ontology (нормативная правда), read goes to indexed cache (рантайм-скорость). При расхождении правда — за Turtle, индекс перестраивается.

В RAGRAF индекс собирается:
- автоматически при `regulation_store.save()` (полная замена триггеров регламента);
- при сидинге фикстур (`_seed_from_fixtures_if_empty`);
- через идемпотентный backfill `_backfill_triggers_if_missing()` для старых регламентов без `:hasTrigger`.

## 7. Событийная архитектура и «сквозная правда без потери»

Сигма — **event-driven** платформа. Это не лозунг, это конкретное архитектурное требование: каждое значимое изменение во внешнем мире (показание датчика, видеособытие, телефонный звонок 112, превышение ПДК) должно превратиться в **событие на шине**, которое матчится против корпуса регламентов и порождает **вердикт + действие**. Цепочка:

```
[Физический мир] → [Датчик/детектор] → [Событие в шине] → [Регламент] → [Вердикт + Действие]
```

Чтобы эта цепочка работала надёжно, в каждой её точке должна быть **сквозная правда** — единый набор identifier'ов, который протягивается без потери от датчика до решения. Без триггеров эта правда обрывается на стыке «Событие → Регламент»:

| Звено                 | Что есть до триггеров                | Что появилось после                              |
| --------------------- | ------------------------------------ | ------------------------------------------------ |
| Датчик                | `subtype_id` в Sensor Library        | то же                                            |
| Событие в шине        | `type`, `payload` по схеме подтипа   | + `sensor_subtype` как явный identifier          |
| Регламент             | параметры, рекомендация              | + `triggers` — декларация «я слушаю эти датчики»  |
| Стык «событие → реглам.» | **неявная маршрутизация в коде ETL** | **явный SPARQL/SQL по индексу триггеров**     |
| Вердикт               | `{level, regulation_id, recommendation}` | то же + trace через trigger_id для аудита    |

Сквозная правда означает: **из любой точки цепочки можно проследить и вперёд, и назад**. Из датчика — какие регламенты на нём держатся (бэйдж «в N регламентах» в Sensor Library). Из регламента — какие сигналы его кормят (секция «Триггеры» в редакторе). Из события — какой триггер сработал и почему вердикт именно такой. Раньше эта связность ломалась на середине; сейчас она замкнута.

## 8. Обратная совместимость и автодеривация

Старые регламенты (без `:hasTrigger` в Turtle) при загрузке в RAGRAF получают триггеры автоматически — двухуровневый fallback в `services/triggers.py`:

1. **Если у регламента есть `flow.json`** (Rule DSL Flow Editor отрисован) — деривируем из flow-графа:
   - для каждой `input`-ноды берём её `paramRef` → это `param_ref` триггера;
   - если на input привязан `sensor`-узел через `bindsTo` → берём его `sensorSubtype` → это `sensor_subtype` триггера.
2. **Если flow.json ещё не нарисован** — деривируем из параметров (1 триггер на параметр), эвристически подставляя `sensor_subtype` и `event_type` по словарю `PARAM_TO_SUBTYPE_HINT` (например `inletPressure → industrial-pressure`, `pm25Concentration → air-quality-pm`).

Обе ветви — лишь **дефолтная заполненность**. Аналитик в UI «Триггеры» может перепривязать любой триггер к другому датчику, удалить лишний или добавить новый. Значение пользователя всегда приоритетнее эвристики.

Результат на текущем корпусе RAGRAF (11 регламентов):
- Покрытие: 11/11 = 100% (52 триггера в сумме).
- 9 подтипов датчиков связаны хотя бы с одним регламентом.
- Лидеры по «частотности»: `industrial-temp` (3 регл.), `adpi-gsm` (2 регл.), `industrial-flow` (2 регл.).

## 9. Интерфейс аналитика и сквозная визуализация

Триггеры и библиотека датчиков — это не только онтологические декларации в Turtle. Это первоклассные объекты редактирования и **сквозного просмотра** в UI. То, чего исходному ТЗ Сигмы не хватало: **аналитик без программистских навыков должен видеть привязку «датчик ↔ событие ↔ регламент» из любой точки интерфейса, и не должен переключаться между вкладками чтобы понять, что система знает о его регламенте**. Эта секция описывает, как мы это собрали.

### 9.1 Четыре экрана — одно состояние

| Экран                        | Что показывает                                                                                  | Откуда читает                                                              |
| ---------------------------- | ----------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| **Sensor Library** (`/sensors`) | Дерево классов→подтипов; на каждом подтипе **бэйдж «в N регламентах»**                          | `regulation_triggers` (агрегат через `GET /api/sensor-subtypes/_usage`)    |
| **Regulation Editor / «Поля»** (`/regulations/{id}/edit`) | На карточке каждого параметра — **янтарная плашка с привязанным датчиком и типом события**       | `Regulation.triggers` (поле `param_ref` == `Parameter.name`)              |
| **Regulation Editor / «Триггеры»** | CRUD-список триггеров: метка, селект параметра, селект подтипа датчика (с `optgroup` по классам), тип события. Orphan-индикатор красной рамкой если `param_ref` указывает на удалённый параметр | `Regulation.triggers`                                                      |
| **Regulation Editor / «Turtle»** | Сырой OWL с триплами `:hasTrigger`, `a :Trigger`, `:sensorSubtype` — готовый для экспорта в SIGMA-bundle | `regulation_to_turtle(reg)` на лету                                       |
| **Flow Editor** (`/regulations/{id}/flow`) | Sensor-нода (оранжевый кружок) с PropertyPanel: селект «класс + подтип» + **JSON-preview payload** из библиотеки | Sensor Library + `flow.json`                                              |

**Ключевое свойство:** все пять видов **показывают одно и то же состояние**. Изменение в любом из них синхронизируется во все остальные при следующем save — без ручной пересинхронизации, без «забыл обновить в Turtle».

### 9.2 Сквозной сценарий: аналитик работает с регламентом без кода

Реальный путь пользователя — например, для регламента «Прорыв на тепловом вводе»:

```
ШАГ 1. Зашёл в Студию аналитика → сгенерировал черновик регламента
         из текста СП → 5 параметров автоматически извлечены.
         ↓
         Backend: regulation_store.save(reg) → DuckDB.
                 derive_triggers_from_parameters → 5 авто-триггеров
                 с эвристическим sensor_subtype (`inletPressure → industrial-pressure`).

ШАГ 2. Открыл регламент в редакторе → «Поля» (tab='form').
         Видит 5 карточек параметров. Под именем каждого — янтарная плашка:
         ┌─────────────────────────────────────────────────────────┐
         │ ⚡ Датчик: industrial-pressure — Манометр промышленный   │
         │   · 📡 telemetry.pressure                                │
         └─────────────────────────────────────────────────────────┘
         Сразу понятно: на этот параметр подписан датчик X, событие Y.
         Это автоматический результат шага 1 — ничего вручную не делал.

ШАГ 3. Не согласен с автопривязкой — переключился на вкладку «Триггеры».
         Открыл селект «Датчик» в карточке inletPressure → выбрал
         «fiber-vibration» (DAS-вибрация) → save регламента.
         ↓
         Backend: regulation_store.save(reg)
                  → DuckDB regulation_triggers обновлено
                  → regulation_to_turtle(reg) генерирует новый Turtle.

ШАГ 4. Переключился обратно на «Поля» — плашка обновилась:
         ⚡ Датчик: fiber-vibration — DAS-вибрация

         Переключился на «Turtle» — увидел свежие триплы:
         :HeatInletBreach_Trigger_trig-inletPressure
             a :Trigger ;
             :paramRef "inletPressure" ;
             :sensorSubtype "fiber-vibration" ;
             :eventType "telemetry.pressure" .

         Переключился на «Датчики» (Sensor Library) — у подтипа
         fiber-vibration в дереве появился бэйдж +1 (теперь в 2 регл.),
         у industrial-pressure — −1.

ШАГ 5. Открыл Flow Editor для этого регламента → перетащил
         sensor-ноду на канвас → выбрал в PropertyPanel «class=p,
         subtype=industrial-pressure», увидел JSON-skeleton события:
         {
           "description": "<событие от industrial-pressure>",
           "timestamp": "2026-05-17T08:42:11Z",
           "payload": { "value": 4.2, "unit": "атм" }
         }
         → соединил с input-нодой n_pressure_in → save flow.
         ↓
         Backend: PUT /regulations/{id}/flow
                  → save_flow (data/flows/...)
                  → derive_params_from_flow (параметры не изменились)
                  → reconcile_triggers_with_flow:
                       trig-inletPressure.sensor_subtype:
                         fiber-vibration → industrial-pressure 🔄
                  → regulation_store.save(reg, sync_flow=False)

ШАГ 6. Вернулся в Edit. Не делал save вручную — но всё свежее:
         «Поля» → плашка inletPressure обновилась обратно
                  на industrial-pressure.
         «Триггеры» → селект датчика тоже показывает industrial-pressure.
         «Turtle» → :sensorSubtype "industrial-pressure".
         «Датчики» → fiber-vibration снова −1, industrial-pressure +1.
```

В классической Сигме шаг 5 был бы дырой: нарисовал sensor на flow — а в Turtle и в карточке параметра ничего не отразилось, надо вручную правильно вписать. У нас это сводится в одну точку через `reconcile_triggers_with_flow`.

### 9.3 Двусторонняя синхронизация: правда — общая

```
                            ┌─────────────────────┐
                            │ regulation_store    │
                            │ (DuckDB authoritati │
                            │  ve: regulations +  │
                            │  parameters +       │
                            │  regulation_triggers│
                            │ )                   │
                            └──────────┬──────────┘
                                       │
                ┌──────────────┬───────┴──────────┬──────────────────┐
                ▼              ▼                  ▼                  ▼
       PUT /regulations  PUT /regulations    GET /regulations   GET /sensor-subtypes
       /{id} (Edit save) /{id}/flow          /{id} (любой       /{id}/regulations
                         (Flow save)         просмотр)          (reverse-lookup)
                │              │                  │                  │
        Edit/«Поля»      Flow Editor:       Edit/«Turtle»       Sensor Library:
        + плашки          sensor-ноды        — на лету           бэйджи «в N регл.»
        + «Триггеры»     + bindsTo           генерируется
                                              :hasTrigger
```

Ключевая инвариантность: **DuckDB — единственный источник правды**, все экраны её только отображают (read) или коммитят в неё (write). Turtle — производный артефакт; flow.json — производный артефакт. Это значит, что **расхождений между видами быть не может by design** — все они смотрят в одну таблицу.

### 9.4 Приоритеты при конфликте: ручной ввод против автоматики

Без чёткого правила приоритета двусторонняя синхронизация ломалась бы первой же правкой. Мы зафиксировали следующие приоритеты в `reconcile_triggers_with_flow`:

| Что изменилось              | Что побеждает                                                                       |
| --------------------------- | ----------------------------------------------------------------------------------- |
| Sensor явно привязан в flow (`sensor.bindsTo`) | **Flow побеждает** — `sensor_subtype` триггера перепишется.                |
| Sensor в flow отсутствует, в триггере выбран руками | **Edit/«Триггеры» побеждает** — ручной выбор сохраняется.        |
| `event_type` / `description` / `label` (не auto) | **Edit/«Триггеры» побеждает** — flow никогда не трогает эти поля.            |
| Параметр удалён из flow и из parameters | Триггер становится **orphan** и удаляется при reconcile (зачистка).        |
| Параметр удалён из flow, но остался в parameters | Триггер **сохраняется** — был создан вне flow, не наш повод его удалять.   |

Это даёт аналитику простой ментальный контракт: «всё, что я ввёл руками, не затрётся при правке flow; всё, что я выразил визуально через sensor.bindsTo, переходит в декларацию автоматически».

### 9.5 Видимая защита от ошибок

UI подсвечивает три случая, которые без визуализации легко пропустить:

1. **Orphan-триггер** — `param_ref` указывает на удалённый параметр. Карточка триггера в «Триггеры» получает красную рамку и бэйдж «параметр не найден». В селекте параметра orphan-значение всё ещё видно с пометкой `(orphan)`, чтобы пользователь его узнал и либо перепривязал, либо удалил.
2. **Триггер без датчика** — `sensor_subtype = null`. На карточке параметра плашка серая, не янтарная: «Триггер «trig-X» — датчик не привязан». Это не ошибка, но визуальный сигнал «не доделано».
3. **Параметр без триггера** — для не-placeholder имён (т.е. реальное имя, не `param_N`) показываем мягкую подсказку «Триггер не создан — добавьте в секции ниже». Не блокируем, но напоминаем.

### 9.6 API-поверхность UI

Для полноты — REST-эндпоинты, на которых строится вся эта визуализация:

```
GET    /api/regulations/{id}                  → Regulation (с triggers)
PUT    /api/regulations/{id}                  → save regulation (форма)
PUT    /api/regulations/{id}/flow             → save flow + reconcile parameters & triggers
GET    /api/sensor-subtypes                   → дерево классов → подтипы
GET    /api/sensor-subtypes/_usage            → агрегат subtype_id → N регламентов
GET    /api/sensor-subtypes/{id}/regulations  → конкретные регламенты на этом датчике
GET    /api/sensor-schemas/{subtype_id}       → поля payload подтипа (для JSON-preview)
```

Каждый из этих маршрутов отдаёт **полную картину для своего экрана**, без N+1 запросов и без необходимости заводить промежуточный middleware.

## 10. Композиция регламентов через события (meta-rules)

В исходном ТЗ Сигмы регламент — атомарная единица: на вход параметры с датчиков, на выход вердикт. На практике в город-/промышленных доменах встречаются сценарии, где **выход одного регламента становится событием для другого**:

- **Каскадная эскалация.** Регламент A («прорыв на тепловвводе») сработал → выдал `smart_valve_close` с `priority=1`. Регламент B («массовое отключение в районе») должен агрегировать такие срабатывания и эскалировать в ЕДДС, если их 5+ за час.
- **Корреляция критических событий.** Регламент C («комплексная аварийная ситуация») активируется когда **одновременно** сработали A («утечка газа») И D («пожарная сигнализация»). Это «событие от регламента», а не от датчика.
- **Двухступенчатые рекомендации.** Регламент A выдал «запросить обходчика». Через 10 минут обходчик не подтвердил — регламент E («автоматическое перекрытие при пропуске подтверждения») реагирует на таймаут как на событие.

### 10.1 Модель — два поля в `RegulationTrigger`

К существующим полям (`sensor_subtype`, `event_type`) добавлены ещё два:

```python
class RegulationTrigger:
    # ... id, label, param_ref, sensor_subtype, event_type ...
    source_regulation: str | None = None   # FK на Regulation.id
    source_output: str | None = None       # action из output-ноды регламента-источника
```

**Взаимоисключимость.** Триггер с одним и тем же `id` либо «слушает датчик» (`sensor_subtype`), либо «слушает выход регламента» (`source_regulation`), либо «ловится по типу события» (`event_type` без других якорей). UI секции «Триггеры» реализует это сегментным переключателем «Датчик / Регламент / —» — при переключении противоположные поля сбрасываются.

**`source_output` опционален**: если задан — триггер активируется только конкретным action'ом (например `smart_valve_close`); если `null` — любым вердиктом регламента-источника.

### 10.2 Turtle-сериализация

```turtle
:MassOutageRegulation
    a :Regulation ;
    :name "Регламент массового отключения" ;
    :valveClosedCount 5.0 ;   :valveClosedCountDeviation 0.0 ;
    :hasTrigger :MassOutage_Trigger_trig-valveClosedCount .

:MassOutage_Trigger_trig-valveClosedCount
    a :Trigger ;
    rdfs:label          "Закрытие клапанов другими регламентами"@ru ;
    :paramRef           "valveClosedCount" ;
    :sourceRegulation   "heat-inlet-breach" ;
    :sourceOutput       "smart_valve_close" .
```

Apache Jena парсит это нативно (две новых DatatypeProperty в дополнение к существующим). При импорте SIGMA-bundle парсер `parse_regulation_turtle` восстанавливает оба поля — round-trip полный.

### 10.3 Reverse-lookup и видимость

Аналогично датчикам, у регламента-источника появляется бэйдж «является триггером для N других регламентов». UI его отображает:

- **В Edit/«Поля»** — фиолетовая плашка в начале формы со списком регламентов-потребителей и action'ов, на которые они подписаны. Каждый пункт — `<Link>` на регламент-потребитель.
- **В Graph View** — новый тип ребра между двумя `Regulation`-узлами: пунктирная фиолетовая линия с подписью `triggers_regulation:<action>`. Композиционные связи визуально отличаются от структурных (parameter / constraint / recommendation).

REST для этих видов:
```
GET /api/regulations/{id}/triggered-by    → список регламентов, слушающих этот
GET /api/regulations/{id}/output-actions  → список action'ов из flow.json (для селекта)
```

### 10.4 Защита от циклов

Сейчас UI не позволяет триггеру ссылаться на **тот же регламент** (self-loop) — селект `source_regulation` исключает текущий ID. Кросс-циклы (A→B→A) пока **не детектятся** на уровне сохранения: для них нужен топологический обход графа триггеров. Это в roadmap — на текущей фазе достаточно self-loop-защиты, потому что число регламентов невелико и аналитик увидит цикл визуально в Graph View.

### 10.5 Runtime: что делается сейчас, что для CORE Сигмы

На стороне RAGRAF композиция — **декларативная**. Output-ноды flow_executor возвращают `{level, recommendation, action}` в HTTP-ответе, **не публикуют событие в шину**. Это означает: декларация триггера готова, но цепочка A→B на рантайме не замыкается без CORE Сигмы.

Это намеренное разделение зон ответственности:
- **RAGRAF** — author/model layer: описывает «что должно произойти». Триггер с `source_regulation` — это онтологический контракт «регламент B подписан на регламент A».
- **CORE Сигма** — execute layer: реализует «как именно». Для замыкания цепочки CORE должен:
  1. После каждого вердикта регламента A публиковать событие вида `{event: "regulation.output", regulation_id: "A", action: "smart_valve_close", level: 1, …}` в шину.
  2. ETL-приёмник матчит это событие против индекса `regulation_triggers` (WHERE source_regulation = 'A' AND (source_output IS NULL OR source_output = 'smart_valve_close')) — тот же O(1)-SQL что и для датчиков.
  3. Найденный регламент B исполняется с этим событием как одним из входов.

Декларация в Turtle уже сейчас готова к этому переходу: CORE при импорте SIGMA-bundle получает полный граф композиции и может маршрутизировать события без переделок.

### 10.6 Что появилось в UI

| Экран                                         | Что видно про композицию                                          |
| --------------------------------------------- | ----------------------------------------------------------------- |
| **Edit / «Поля»**                              | Фиолетовая плашка-баннер «является триггером для N» сверху формы. На карточке параметра плашка триггера тоже фиолетовая (вместо янтарной), если источник — другой регламент. |
| **Edit / «Триггеры»**                          | Сегментный переключатель «Датчик / Регламент / —». При выборе «Регламент» — два каскадных селекта: «Регламент-источник» (с исключением self-loop) и «Output (action)» (подгружается из flow.json выбранного регламента). |

| **Граф связей**                                | Пунктирное фиолетовое ребро между `Regulation`-узлами с подписью `triggers_regulation:<action>`. Визуально отличает композиционные связи от структурных. |

Аналитик в **любой** точке (плашка, селект, Turtle, Graph) видит одно и то же состояние композиции — синхронизация по тому же контракту, что и для триггеров на датчики.

## 11. Связь с SIGMA-bundle и round-trip

При экспорте регламента в SIGMA-bundle (`/api/regulations/{id}/export-bundle` → ZIP с `data.ttl` + `shapes.ttl` + `manifest.json`) триггеры выгружаются как часть `data.ttl` через `:hasTrigger`. Apache Jena парсит это нативно. При обратном импорте (`/api/sigma-import/bundle`) парсер `parse_regulation_turtle` восстанавливает `triggers` из триплов — round-trip полный.

Это значит: **если СИГМА работает с регламентом, который пришёл из RAGRAF, она получает декларацию триггеров «как есть»**, без необходимости отдельно их пересинхронизировать.

---

## 12. Что меняется на стороне СИГМЫ (требования к ETL/CORE)

Внедрение триггеров требует от рантайма СИГМЫ всего двух вещей, обе минимальные:

1. **На приём события**: вместе с `type` и `payload` принимать поле **`sensor_subtype`** (опционально — `event_type`). Если эти поля не передаются — fallback на старую маршрутизацию по `type`, регресс не возникает.
2. **На матчинг события → регламент**: выполнить SPARQL по `:hasTrigger`/`:sensorSubtype` (или эквивалентный SQL по `regulation_triggers`-индексу, если индекс реплицирован в производственный стор СИГМЫ). Если в графе нет ни одного триггера на конкретный `sensor_subtype` — fallback на старый маршрут по `type` срабатывает автоматически.

То есть для команды СИГМЫ интеграция **аддитивна** — ничего не ломается, добавляется новое поле + новый SPARQL.

## 13. Сводка: что мы сделали и почему

| Вопрос                                                          | Ответ                                                                                  |
| --------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Какую идею из исходного ТЗ мы развили?                          | «Регламент описывает себя сам в OWL/Turtle + валидируется SHACL»                       |
| Что в этой идее не дотягивало до event-driven продукта?         | Отсутствовала декларация «какие события активируют регламент» — была неявная маршрутизация по типу датчика |
| Что мы добавили?                                                 | Класс `:Trigger`, отношение `:hasTrigger`, поле `triggers: list[RegulationTrigger]` в `Regulation`, DuckDB-индекс `regulation_triggers`, UI-секция, reverse-lookup endpoint    |
| Почему именно так?                                              | Триггер — отдельная сущность с уникальной identity; один регламент = N триггеров (множественность входов решена); двухслойное хранение (Turtle = правда, DuckDB = индекс) дает онтологическую чистоту + O(1) рантайм |
| Что это даёт продукту?                                          | Сквозная правда от датчика до вердикта без потери; маршрутизация события масштабируется на 10⁴ регламентов; **аналитик работает с регламентом без кода — пять экранов (Sensor Library / Поля / Триггеры / Turtle / Flow) показывают одно состояние и синхронизируются автоматически**; reverse-lookup «какие регламенты держатся на этом датчике»; видимая защита от orphan-привязок и недоделанных триггеров |
| Что нужно от команды СИГМЫ?                                     | На приём событий — необязательное поле `sensor_subtype`. На матчинг — добавочный SPARQL по `:hasTrigger`. Старый flow остаётся, регресса нет. |

Триггеры — это не альтернатива онтологическому подходу Сигмы, это его **завершение для event-driven сценариев**. Регламент перестаёт быть просто справочником значений и становится самодостаточным контрактом «я отвечаю на эти события и выдаю этот вердикт».
