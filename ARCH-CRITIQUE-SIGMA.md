# Критическая оценка раздела 6 «Описание архитектуры СИГМА» (2025)

> Документ: «Платформа интеллектуального управления городской средой "СИГМА".
> Описание архитектуры. ИИ01-01-02-61-ЛУ», ЦИИНГУ, НГУ, 2025.
> Утвердил: ректор М.П. Фёдорук. Согласовали: А.Н. Люлько, И.А. Болдырев, А.Н. Ряскин.

Цель — оценить **§ 6 «Диаграмма последовательности движения события»** через
призму того, что мы построили в RAGRAF, и предложить корректировки.

## TL;DR

В § 6 шаг 6 («CORE → Регламенты/RAG: запрос применимых регламентов») —
**узкое место архитектуры на корпусе 10 000+ регламентов**. Использование
RAG как механизма маршрутизации события путает две роли (семантический
поиск помощника vs. event-driven диспетчер). RAGRAF решает это через
**trigger-based индексированный реверс-lookup за O(1)** — паттерн, который
напрямую переносится в СИГМА без слома их событийной модели.

Дополнительные пробелы у СИГМА: отсутствие композиционных цепочек между
регламентами, отсутствие трёхзначной логики (Kleene unknown), смешение RAG-
помощника и event-router. Всё это в RAGRAF уже сделано.

## Узкое место №1: маршрутизация через RAG

### Что в СИГМА (§ 6, строки 6-7)

```
6  CORE → Регламенты / RAG:  Запрос применимых цифровых регламентов
                              и целевого состояния
7  Регламенты / RAG → CORE:  Возврат правил, порогов и основания проверки
```

То есть: на **каждом** входящем событии CORE обращается в подсистему
«Регламенты / RAG» для поиска **применимых** правил.

### В чём проблема

RAG (Retrieval Augmented Generation) хорош для:
- семантического поиска по неструктурированному корпусу;
- Q&A пользователя над документами;
- помощи аналитику при анализе текста.

RAG **категорически плох** для маршрутизации событий в real-time:

| Метрика | RAG (semantic search + LLM) | Trigger-based индекс |
|---|---|---|
| Latency на запрос | 100-500 мс (embedding + ANN + LLM re-rank) | < 1 мс (B-tree lookup) |
| Throughput при N=10 000 регламентов | ~10-50 событий/сек | > 10 000 событий/сек |
| Детерминированность | Стохастическая (порядок чанков, score threshold) | Полностью детерминирована |
| Объяснимость («почему именно эти регламенты?») | Низкая («similarity 0.78») | Высокая («match sensor_subtype=industrial-pressure») |
| Стоимость | LLM-токены + vector DB | DuckDB-индекс (free) |
| False negatives при пограничных формулировках | Возможны | Невозможны (точный equality) |

На корпусе 10 000 регламентов и потоке 100 событий/сек СИГМА упрётся
в LLM-затраты (~$15 000/мес если использовать OpenAI-class) и в латентность
(если регулярно срывается SLA «решение оператору за 30 сек»).

### Как сделано в RAGRAF

Каждый регламент **сам декларирует** какие события он слушает. Декларация
живёт в трёх местах одновременно (источник правды один):

**1. Turtle data.ttl** — каноническая сериализация для интеграции с СИГМА:
```turtle
:HeatInletBreachRegulation a :Regulation ;
    :hasTrigger :HeatInletBreach_Trigger_trig-inletPressure .

:HeatInletBreach_Trigger_trig-inletPressure a :Trigger ;
    rdfs:label "Давление на входе"@ru ;
    :paramRef "inletPressure" ;
    :sensorSubtype "industrial-pressure" ;
    :eventType "telemetry.pressure" .
```

**2. DuckDB-таблица `regulation_triggers`** с индексами:
```sql
CREATE TABLE regulation_triggers (
    source_id      VARCHAR NOT NULL,
    trigger_id     VARCHAR NOT NULL,
    param_ref      VARCHAR NOT NULL,
    sensor_subtype VARCHAR,
    event_type     VARCHAR,
    source_regulation VARCHAR,  -- композиция между регламентами
    source_output     VARCHAR,
    PRIMARY KEY (source_id, trigger_id)
);
CREATE INDEX idx_trig_subtype ON regulation_triggers(sensor_subtype);
CREATE INDEX idx_trig_event   ON regulation_triggers(event_type);
CREATE INDEX idx_trig_source_reg ON regulation_triggers(source_regulation);
```
Источник: [`backend/app/services/regulation_store.py:128-160`](backend/app/services/regulation_store.py)

**3. Flow.json:** `sensor.bindsTo → input.paramRef` — то же самое, но в
графической модели для редактора.

### Поток обработки события в RAGRAF (предлагается СИГМА для замены § 6 шаг 6-7)

```
Event{sensor_id="kpd-12", sensor_subtype="industrial-pressure", value=23.5, ts=...}
   ↓
[CORE: lookup_regulations_by_subtype()]
   → SELECT source_id FROM regulation_triggers
     WHERE sensor_subtype = 'industrial-pressure'
   → возвращает 3-5 регламентов за < 1 мс
   ↓
[CORE: parallel evaluate each regulation against event.value]
   → для каждого: input → threshold → compare → formula → output
   → результат: уровень + recommendation + fired_path
```

Бэкенд-эндпоинт уже есть: `POST /api/regulations/{id}/execute` —
см. [`backend/app/services/flow_executor.py`](backend/app/services/flow_executor.py).

Reverse-lookup для UI: «какие регламенты слушают этот датчик»:
- API: `GET /api/sensor-subtypes/{subtype_id}/regulations`
- Реализация: [`regulation_store.py:list_by_sensor_subtype()`](backend/app/services/regulation_store.py)

## Узкое место №2: нет композиции между регламентами

### Что в СИГМА (§ 6)

Линейный сценарий: «событие → CORE → один регламент → рекомендация». Нет
механизма «output регламента A триггерит регламент B». Между тем 80%
реальных кейсов МЧС / ЕДДС / городской безопасности — **двух-трёхуровневые**:

```
Звонок 112 → классификация
   ↓ (если массовое)
Перевод ЕДДС в режим повышенной готовности
   ↓ (если региональная ЧС)
Доклад в ЦУКС ГУ МЧС
   ↓ (параллельно при пожаре)
Эскалация в АДПИ-мониторинг
```

В архитектурном документе СИГМА это не представлено вообще.

### Как сделано в RAGRAF

Поле `RegulationTrigger.source_regulation` указывает: «слушать output другого
регламента, не sensor». См. `regulation_triggers.source_regulation` + индекс
`idx_trig_source_reg`. Это даёт детерминированные event-driven цепочки между
регламентами **без RAG-обращения**.

Реальный пример работает прямо сейчас — 4 регламента ЕДДС р.п. Кольцово
связаны в цепочку (call-112 → mode-escalation → cuks-report + adpi-monitoring):
[`backend/data/fixtures/koltsovo-edds-*.flow.json`](backend/data/fixtures/).

Визуализация: Graph View → фиолетовые рёбра `triggers_regulation:<action>`,
см. README раздел «5. Rule Flow Editor».

**Перенос в СИГМА:** в схеме событий добавить тип «событие-вердикт» с
полями `{source_regulation_id, output_action, level}`. CORE при сохранении
вердикта регламента A автоматически инжектит такое событие в ETL → запускается
шаг 6 с lookup'ом по `source_regulation` индексу → найден регламент B.

## Узкое место №3: смешение RAG-помощника и event-router

§ 4 СИГМА говорит:

> **RAG / поиск по регламентам** — Поиск применимых правил и документов,
> поддержка пользователя при анализе регламентов и подготовке ответов.

Это **две разные задачи**, объединённые под одной компонентой:

| Задача | Подходящий механизм |
|---|---|
| Оператор СЦ задаёт вопрос «что делать при разрыве трубы?» | RAG / семантический поиск по корпусу |
| Прилетает event{sensor=industrial-pressure, value=23}, нужен applicable regulation | Trigger-based индекс |

В RAGRAF мы их разделили:
- **RAGU-интеграция** ([`backend/app/services/ragu_service.py`](backend/app/services/ragu_service.py))
  для семантического Q&A по корпусу — это поддержка АНАЛИТИКА в Студии аналитика.
- **regulation_triggers + indexes** для маршрутизации событий — это работа
  RUNTIME executor'а.

Это разделение **обязательно** для production. Рекомендация СИГМА: переименовать
§ 4 компоненту в «RAG / поиск по корпусу для Q&A» и **добавить отдельный
компонент «Event Router (trigger index)»** между ETL и CORE.

## Узкое место №4: классическая булева логика → false negatives

§ 6 шаг 8 «CORE → CORE: Оценка критичности, выявление отклонения». Если
sensor offline (event просто не прилетел), CORE не выявит отклонения и
выдаст «всё ок» — это false negative.

В RAGRAF (фикс 043, см. [README § 8.1 «Опыт математиков»](README.md))
введена трёхзначная логика Клини: result ∈ {true, false, **unknown**}.
При offline-датчике formula возвращает `unknown`, в trace явная пометка
«нет данных: pressure», уровень не выдаётся (level=0 не равно «всё ок»,
а «недостоверно»).

**Перенос в СИГМА:** в схеме события добавить статус `evidence_level ∈
{measured, derived, unknown}`. В шаге 8 правило: «если хотя бы один
обязательный параметр события — unknown, итоговый статус = "недостоверно"
с явной пометкой в карточке руководителю».

## Узкое место №5: «единая база событий» — один store на все стадии

§ 8 СИГМА выделяет 4 типа данных: исходные / нормализованные / обогащённые
события + регламенты. Но это **логическое** разделение в одной БД.

На потоке 100 событий/сек это **возможно** в Postgres с правильным
партиционированием, но **рискованно**:
- read-heavy запросы дашборда руководителя конкурируют с write-heavy ETL;
- горячий стол нормализованных событий душит запросы по регламентам;
- журналы аудита растут со скоростью x4 от потока.

**Рекомендация СИГМА:**
- Сырые/нормализованные события → time-series store (Clickhouse / TimescaleDB)
  с TTL 90 дней.
- Регламенты → реляционный store с MVCC (Postgres).
- Аудит-цепочки → отдельная append-only БД (Postgres logical-replicated в archive).
- Кэш «текущее состояние объектов» → Redis.

В RAGRAF на текущей фазе всё в одной DuckDB — это сознательное упрощение
для прототипа (cm. [ARC.md «Слой хранения»](ARC.md)). Когда продукт пойдёт
в production на 10k регламентов, разделение хранилищ предусмотрено
в roadmap'е.

## Что в СИГМА лучше чем у нас (честно)

1. **Реестр модулей с паспортом** (§ 4 «Реестр модулей»). У нас регламенты
   живут как фикстуры без явного паспорта. Полезно для масштабирования
   экосистемы. → план: добавить поле `module_passport` к каждому regulation_store.
2. **API Gateway как отдельный компонент** (§ 4). У нас FastAPI обрабатывает
   всё в одном app. Для production-инстанса с rate-limiting и API-key auth
   нужен gateway (Kong / Tyk / nginx). → план: на стадии production.
3. **Чёткое разделение пилот / тест / прод контуров** (§ 9). У нас один
   Railway-инстанс. → план: на стадии production.
4. **Аудит-цепочка «событие → регламент → рекомендация → действие пользователя →
   результат»** (§ 2 «Объяснимость»). У нас есть `regulation_history`
   (snapshots регламента), но цепочка действий оператора не записывается.
   → план: добавить таблицу `incident_audit_log` с FK на event_id, regulation_id,
   user_action_id, status.

## Минимальные правки в § 6 СИГМА

Предлагаю следующие конкретные изменения:

### Шаг 6 — заменить
```
было: CORE → Регламенты/RAG: Запрос применимых цифровых регламентов
стало: CORE → Event Router (Trigger Index): Получить список регламентов,
       у которых хотя бы один Trigger.sensor_subtype или Trigger.event_type
       совпадает с атрибутами входящего события (O(1) индекс-lookup).
```

### Шаг 7 — заменить
```
было: Регламенты/RAG → CORE: Возврат правил, порогов, основания проверки
стало: Event Router → CORE: Возврат [regulation_id], для каждого CORE
       подгружает RuleDSL (flow.json + параметры) из контура регламентов.
       Сам контур регламентов остаётся для batch-операций (создание/
       редактирование), не на критическом пути обработки события.
```

### Добавить шаг 8a (между 8 и 9)
```
CORE → Event Router (Trigger Index): Если оценка дала вердикт уровня ≥ 1
       и регламент имеет output-action, инжектировать событие-вердикт
       {source_regulation_id, output_action, level} обратно в ETL.
       (Это даёт композиционные цепочки эскалации.)
```

### Добавить в шаг 8 — Kleene unknown
```
было: CORE → CORE: Оценка критичности, выявление отклонения
стало: CORE → CORE: Оценка критичности по трёхзначной логике
       {measured, derived, unknown}. При отсутствии хотя бы одного
       обязательного параметра события — статус «недостоверно»
       с пометкой в карточке руководителю; вердикт уровня НЕ выдаётся.
```

## Прогноз эффекта

После этих правок СИГМА:
1. Снижает SLA-латентность маршрутизации с ~250 мс (RAG) до < 5 мс (index).
2. Получает реальную поддержку 10k+ регламентов и > 1000 событий/сек.
3. Закрывает класс ошибок «уверенный false при offline-датчике».
4. Получает event-driven композицию (МЧС → ЕДДС → ЦУКС) без переписывания CORE.
5. RAG остаётся, но в правильной роли — Q&A над корпусом для оператора СЦ.

Всё это в RAGRAF уже работает и проверено (10 регламентов кампуса НГУ +
4 регламента ЕДДС Кольцово, 0 ошибок валидации). Перенос архитектурного
паттерна занимает 2-3 человеко-недели на стороне СИГМА.
